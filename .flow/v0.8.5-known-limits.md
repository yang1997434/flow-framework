# v0.8.5 known limits — deferred to v0.8.6 backlog

> **Source**: codex R2 review, decided not-must-fix in this release.
> **Owner**: v0.8.6 planning.
> **Date noted**: 2026-05-09.

## I3-B — `_collect_untracked()` collapses nested files into directory entry

`git status --porcelain` reports an untracked directory as a single
`?? dir/` line, not per-file. Our untracked collector
(`scripts/common/diff_summary.py::_collect_untracked`) consequently
lists only the parent directory entry; per-file structural-map entries
for files inside a brand-new directory are degraded.

* Symptom: when a Round N implementer creates `src/new_module/foo.py`
  + `src/new_module/bar.py` inside a fresh `src/new_module/`
  directory, the Round N+1 prompt's diff map lists
  `src/new_module/` as one entry instead of two file entries.
* Severity: low — the directory name still surfaces; the exact
  files are recoverable from the worktree.
* Workaround: run `git add -N` on untracked files at the end of an
  implementer round (would promote them to staged-but-no-content,
  which `_collect_staged` would pick up per-file).
* Fix candidate (v0.8.6): use `git ls-files --others --exclude-standard`
  with explicit per-file enumeration instead of porcelain status.

## I5 — `auto_dispatch_task` lacks fail-closed unknown-kwarg test

`auto_dispatch_task` accepts `telemetry_emit_fn` as an explicit named
parameter (so a misspelt call DOES raise `TypeError` at runtime). What
v0.8.5 lacks is a test fixture that exercises the misspelt call path —
unlike the prompt-builder kwarg test
(`tests/unit/test_dispatch_template_v085_enrichment.py::FailClosedKwargAssertion`).

* Severity: low — runtime fail-closed behaviour is provided by
  Python's parameter binding; the gap is purely test introspection.
* Risk: a future caller that accidentally passes
  `telemetry_emit_fn=None` and later mistypes the kwarg name would
  silently no-op the telemetry. The risk surface is identical for
  every other kwarg in the codebase that lacks a corresponding
  introspection test, so this is not specific to v0.8.5.
* Fix candidate (v0.8.6): mirror the prompt-builder pattern —
  `tests/unit/test_dispatch_template_v085_enrichment.py::FailClosedKwargAssertion`
  asserts `with assertRaises(TypeError): build_implementer_prompt(
  task_brief="x", prev_round_diff_summarie=...)`. Add the same
  pattern for `auto_dispatch_task` + `GateRunner.__init__`.

## I3-related — same-file-multiple-sources renders duplicate hunk blocks

Discovered while writing the v0.8.5 codex-R2 I6 production-path test
(`tests/smoke/test_v085_production_path.py::TestRound2EnrichmentViaFullRetryLoop`).

* Symptom: when a single file appears in multiple `_collect_*`
  sources (e.g. `a.py` is BOTH committed AND has unstaged
  modifications), the rendered diff map lists the file's hunk header
  block twice. The stat block correctly shows two stat lines (one
  per source state — that's intentional and is what I3-A's fix
  preserves), but `_render` then iterates the stat list and emits
  the per-file hunk block once per stat-list occurrence.
* Severity: cosmetic. The diff-map enrichment is auxiliary context
  (PRD R4 — "structural map only; no code content"); duplicated
  hunk headers don't mislead the implementer (they're literally the
  same lines). The total-line cap (200) still binds.
* Why deferred: the fix requires refactoring `_render` to dedupe
  the hunk-block iteration without losing stat-block fidelity, and
  is mechanically distinct from I3-A's `_collect_unstaged` fix.
  Caught by the production-path test but not asserted on (the
  assertions specifically allow at-most-N stat lines per file but
  don't pin hunk-block counts).
* Fix candidate (v0.8.6): make `_render` iterate `hunk_map.keys()`
  separately from the stat list (or dedupe stat-iteration before
  emitting hunk blocks), with merged hunk lists per path.

All three items are tracked for v0.8.6 RFC scoping. None gate the
v0.8.5 ship since they are low-severity / cosmetic / test-coverage gaps.
