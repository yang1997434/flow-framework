# v0.8.1 Execution Semantics

**Scope**: protocol / runtime semantics *between* the v0.8.1 primitives. NOT a re-design of the primitives themselves — those live in `prd.md §1`. This doc fills the seam codex round-2 + consult flagged: PRD lists `worktree-per-task`, `acceptance gate`, `budget`, `notification`, `AFK timeout`, `staleness` as primitives, but does not define how they compose, especially in failure paths.

**Why this layer exists**: `prd.md §3` v0.8.1 sketch is a 12-bullet feature list. Each bullet is sound in isolation. The failure modes are at the joins (e.g., budget exhaustion *during* an e2e acceptance run). Implementation will pick an arbitrary policy if the design doesn't pin one — which is exactly the silent-failure mode codex round 2 RED'd against.

**Output discipline**: decision tables, not essays. Every row resolves to one of `DECIDED <answer>` / `OPEN <question + options>` / `DEFERRED <to ship>`. No pure prose between tables.

**Ship gate**: codex round 3 reviews this doc + the v0.8.1 plan derived from it. GREEN required.

## §1 Stop semantics — block / abort / wait disambiguation

**Question**: for each failure trigger in v0.8.1, which of `{block, abort, wait}` fires, and what state transition follows?

**Definitions (DECIDED)**:
- `block` = transient pause, write `blocked.md` + Tier 2 notification, await user choice. Worktree preserved. Resume protocol applies.
- `abort` = terminal stop for this task. Worktree marked `aborted/`, partial diff preserved (NOT discarded), task marked failed in progress.md. NO resume — must restart task or escalate to interactive.
- `wait` = silent pause, no notification, polling loop. Used only for AFK-timeout `wait` mode.

| # | Trigger | Action | Worktree state | progress.md task status | Notification |
|---|---------|--------|----------------|-------------------------|--------------|
| 1 | budget hit (max_files / max_deps / max_retry) | `block` | preserved | `blocked` | Tier 1+2 |
| 2 | budget hit (max_elapsed_min) mid-task | `block` | preserved | `blocked` | Tier 1+2 |
| 3 | manifest violation (file outside scope.allowed) | `block` | preserved (diff frozen) | `blocked` | Tier 1+2 |
| 4 | manifest violation (untracked file outside scope) | `block` | preserved | `blocked` | Tier 1+2 |
| 5 | acceptance criterion fail (cmd / file / json / http) | `block` | preserved | `blocked` | Tier 1+2 |
| 6 | acceptance criterion fail (e2e) | `block`; **escalate flag** (PRD §1.3 says e2e fail must escalate) | preserved | `blocked_escalate` | Tier 1+2 |
| 7 | baseline test newly fail | `block` | preserved | `blocked` | Tier 1+2 |
| 8 | staleness trigger fires mid-task | `block` | preserved | `blocked_stale` | Tier 1+2 |
| 9 | codex review RED (per-task) — retry budget OR `max_codex_rounds_per_task` exhausted (DECIDED Q2.2 + Q3.2) | `block` (enter retry whitelist flow) | preserved | `blocked_review` | Tier 1+2 |
| 10 | codex round-3 RED on plan | not a runtime trigger — plan-time gate; `abort` ship cycle | n/a | n/a | n/a |
| 11 | AFK timeout, contract `afk_on_timeout: wait` | `wait` (poll); **`max_elapsed_min` clock paused** (DECIDED Q2.1) | preserved | `waiting` | none (silent) |
| 12 | AFK timeout, contract `afk_on_timeout: abort` | `abort` | aborted/ | `aborted_afk` | Tier 1+2 (final) |
| 13 | irreversible action attempted by subagent | `block` (always; user-sovereignty escape) | preserved | `blocked_irreversible` | Tier 1+2 |
| 14 | unforeseen high-reversal fork (3rd stop condition) | `block` | preserved | `blocked_fork` | Tier 1+2 |
| 15 | spec/scope change detected (1st stop condition) | `block` | preserved | `blocked_spec` | Tier 1+2 |
| 16 | nested-autonomy attempt (subagent tries auto-execute) | `abort` (DECIDED Q1.3 — pulled from v0.8.2 to v0.8.1) | aborted/ | `aborted_nested` | Tier 1+2 |
| 17 | `auto_prepare_interrupted` — pre-`auto_engaged` crash with auto contract loaded (DECIDED R10) | `block` with `block_review_state` | preserved if existed | `blocked_auto_prepare` | Tier 1+2 |
| 18 | `post_merge_verify_failed` — local merge applied but post-merge integration verify failed (DECIDED R3+R4) | `block` with merge intact (NO auto-revert) | n/a (worktree merged) | `blocked_post_merge` | Tier 1+2 |

**DECIDED (round 1, 2026-05-06)**:
- `Q1.1` → **DECIDED YES**. Row 6 `blocked_escalate` reduces user choices to `{abort, interactive, split}` (no `extend retry`). Rationale: PRD §1.3 says e2e/behavior fail must escalate; offering retry contradicts escalation.
- `Q1.2` → **DECIDED YES**. AFK `wait` mode caps at 24h then auto-converts to `block` (Tier 1+2 fires). Rationale: avoids worktree disk-occupy + ambiguous "still alive" state.
- `Q1.3` → **DECIDED v0.8.1** (was PRD v0.8.2). Nested-autonomy detection ships in v0.8.1. Rationale: deferring opens a window where auto unlocks but nested attack surface opens simultaneously; safety stack must be coherent.

## §2 Budget accounting — what counts against what

**Question**: for each "elapsed time / file count / retry" budget, which activities accrue against it?

**Activities** (taxonomy DECIDED):
- subagent code execution (the tool calls)
- orchestrator manifest verification
- per-task codex review
- acceptance criterion runs (cmd / file / json / http / e2e)
- baseline test run (start-of-task + on resume)
- notification wait time (between block + user response)
- AFK wait time
- worktree setup / merge time

**Clock model (DECIDED Q2.1, round 4 expanded, round-3-fix R10 reinforced)**: `max_elapsed_min` uses **run wall-clock paused during `blocked` / AFK `wait` / crash-downtime / `block_review_state`**. Pauses on `blocked.md` write (any `block_type`); resumes on user resume action. Pauses on AFK `wait` entry; resumes on user activity OR converts to `block` at 24h cap (§1 row 11). Pauses on orchestrator crash (gap between last heartbeat and next-startup detection); resumes on user choice from post-`auto_engaged` crash recovery `block` OR `block_review_state` resolution. **AFK "user activity" defined mechanically (S4)**: any of {terminal stdin activity at orchestrator pid, presence of `flow resume <slug>` invocation, `blocked.md` deletion via `flow resume`}. No active-work-only sub-accounting — predictability wins for the unpaused window.

| Activity | `max_elapsed_min` | `max_files_changed` | `max_new_deps` | `max_retry_per_task` | `max_codex_rounds_per_task` |
|----------|-------------------|---------------------|----------------|----------------------|----------------------------|
| subagent code execution | ✅ counts | ✅ counts | ✅ counts | n/a | n/a |
| orchestrator verification | ❌ excluded (overhead) | n/a | n/a | n/a | n/a |
| per-task codex review (round) | ❌ excluded (out-of-band) | n/a | n/a | n/a | ✅ +1 per round |
| acceptance cmd / file / json / http | ✅ counts | n/a | n/a | n/a | n/a |
| acceptance e2e | ✅ counts (with cap) | n/a | n/a | n/a | n/a |
| baseline test | ❌ excluded (must run, not work) | n/a | n/a | n/a | n/a |
| `blocked` notification wait | ⏸ **PAUSED** (DECIDED Q2.1) | n/a | n/a | n/a | n/a |
| AFK `wait` time | ⏸ **PAUSED** (DECIDED Q2.1) | n/a | n/a | n/a | n/a |
| worktree setup / merge | ❌ excluded (overhead) | n/a | n/a | n/a | n/a |
| retry attempt (full re-run after RED) | ✅ counts (each attempt) | ✅ counts (cumulative) | ✅ counts (cumulative) | ✅ +1 per attempt | n/a |
| review-issue `rejected_with_rationale` | n/a | n/a | n/a | ❌ does NOT consume (DECIDED Q3.2) | ✅ counts (DECIDED Q3.2) |
| post-merge verification (R4) | ✅ counts (with timeout cap) | n/a (post-merge diff already counted) | n/a | n/a | n/a |
| crash downtime (gap to next-startup detection) | ⏸ **PAUSED** (R10 reinforced) | n/a | n/a | n/a | n/a |

**Edge cases (DECIDED)**:
- e2e acceptance run that exceeds `max_elapsed_min` → e2e is `aborted` (not `failed`), criterion marked `inconclusive`, then row §1.5 fires (criterion fail → block). Distinguish in `acceptance-issues.jsonl`.
- Budget hit *during* acceptance gate → block at the criterion boundary, not mid-criterion. If a criterion is mid-run, let it finish, THEN block before next criterion.
- `max_files_changed` measured at orchestrator-derived diff time (post-task, not during). Subagent advisory self-report is NOT consulted.

**DECIDED (round 2, 2026-05-06, codex-refined)**:
- `Q2.1` → **DECIDED wall-clock paused during blocked/AFK-wait**. Codex consult refined Claude's "wall-clock never paused" because counting user-offline time made §1 row 11 + 24h cap hostile to legit AFK. Pause semantics: clock stops on `blocked.md` write or AFK `wait` entry; resumes on user resume action. 24h AFK cap (§1 Q1.2) still bounds the pause window.
- `Q2.2` → **DECIDED YES, `budget.max_codex_rounds_per_task` field added, default 3**. Independent counter from `max_retry_per_task`. Each codex review round (gate 4) increments. Exhaustion → §1 row 9 fires `blocked_review`. Forward-compat: schema bump not required (additive field; missing → infer default 3).

## §3 Acceptance gate ordering

**Question**: in what order do gates run, and which failures permit a localized fix vs require escalation?

**Gates (DECIDED order, per task — round-3-fix R4 extended 6→8)**:
1. Baseline tests (start-of-task) — establishes pre-existing fail set
2. Subagent task execution
3. Orchestrator manifest verification (file scope)
4. Per-task codex review
5. Acceptance criteria (per the contract list, in declared order) — **Phase 2 only; Phase 3 escalation rules apply for behavior/e2e (DECIDED R2 — see retry-scope below)**
6. Final regression smoke (suite-wide) — Phase 2 worktree-side
7. **Local merge into integration target** (R3 9-step transactional sequence — see §6)
8. **Post-merge verification in ephemeral verification worktree** (R4 — full acceptance + regression smoke unless criterion has `post_merge_skip: true` with rationale; failure → §1 row 18 block with merge intact)

**Per-gate failure handling**:

| Gate | Local fix permitted? | Escalation trigger | Retry counts? |
|------|----------------------|--------------------|----|
| 1 baseline | n/a (pre-existing) | n/a — carry forward as warning | n/a |
| 1 baseline (newly broken) | n/a (block on §1 row 7) | always block | n/a |
| 2 subagent execution | ❌ (controlled by subagent prompt) | n/a — subagent re-prompts itself | n/a |
| 3 manifest violation | ❌ (always block, codex round 2 RED'd advisory enforcement) | always block (§1 rows 3–4) | n/a |
| 4 codex review RED | ✅ (retry whitelist applies) | `max_retry_per_task` exhausted OR churn detected | ✅ each retry +1 |
| 5 acceptance unit/integration | ✅ (localized fix in retry whitelist) | retry exhausted OR criterion mutated by retry | ✅ |
| 5 acceptance behavior (Phase 2) | ✅ (localized fix in retry whitelist) | retry exhausted | ✅ |
| 5 acceptance behavior (Phase 3) | ❌ **never local** (DECIDED R2 — PRD §1.3 says behavior/e2e Phase 3 fail must escalate) | always escalate | ❌ |
| 5 acceptance e2e | ❌ **never local** (PRD: e2e fail must escalate) | always escalate (§1 row 6) | ❌ |
| 5 acceptance regression (suite) | ❌ never local | always escalate | ❌ |
| 6 final regression smoke | ❌ never local | always escalate | ❌ |
| 7 local merge (R3) | ❌ never local | post-merge verify decides; merge stays intact on fail | ❌ |
| 8 post-merge verify (R4) | ❌ never local | always block (§1 row 18) — user choice: keep_and_fix_interactive / revert_merge / split_followup / abort_run | ❌ |

**Retry-whitelist scope (DECIDED)**:
- Whitelist applies ONLY to gates 4 + 5-unit/integration/behavior.
- Whitelist semantic-diff check (per PRD §1.2 step 6) runs against the BEFORE-retry diff vs AFTER-retry diff. Detects: tests skipped / deleted, fixture narrowing, command flag changes that suppress checks. Detected → escalate (do not consume retry budget further).

**DECIDED (round 2, 2026-05-06, codex-refined)**:
- `Q3.1` → **DECIDED inside worktree**. Baseline must validate the actual execution substrate, not main checkout fantasy state. Catches worktree-induced state drift (lockfile diverged, submodule init missing). Cost: extra baseline run per worktree — accepted as v0.8.1 safety cost.
- `Q3.2` → **DECIDED dual-counter**. `rejected_with_rationale`: does NOT consume `max_retry_per_task`, **but DOES consume `max_codex_rounds_per_task`** (codex-refined; prevents review/override churn loop). Mandatory: log to `decisions.jsonl` with `{type: "review_rejected", issue_id, rationale}` for audit. Implementation note: review-issue disposition path is the only path where retry-budget and round-budget diverge.

## §4 Worktree lifecycle

**Question**: when does a worktree get created / reused / discarded? What happens to it on each terminal state?

**State machine (DECIDED)**:

| State | Trigger to enter | Trigger to leave | Cleanup action |
|-------|------------------|------------------|----------------|
| `pending` | task scheduled in plan | dispatch begins | none |
| `active` | dispatch begins | task completes / aborts / blocks | none |
| `blocked` | §1 rows 1–9 / 13–15 / 17 | resume / user abort | preserve at `.claude/worktrees/<slug>+t<n>+<shortsha>/` |
| `waiting` | §1 row 11 (afk wait) | user activity / 24h cap → block | preserve |
| **`merging`** (DECIDED R3 — new transactional state) | gates 1–6 passed; `merge_started` event written | `task_completed` event OR `post_merge_verify_failed` (→ `blocked_post_merge`) OR mid-transaction crash (→ `blocked` via R10 next-startup detection) | none mid-transition; cleanup deferred to terminal state |
| `merged` | gate 7 (local merge) + gate 8 (post-merge verify) BOTH pass + `task_completed` event | n/a (terminal) | **remove worktree + delete branch** AFTER both events written |
| `blocked_post_merge` (DECIDED R3+R4; **S1 fix — global run block, not just task block**) | gate 8 fails after gate 7 already merged | user choice: `keep_and_fix_interactive` / `revert_merge` / `split_followup` / `abort_run` | preserve worktree + verification worktree (forensics); merge intact in main UNTIL user chooses; **all later tasks in the same run halt at `pending`** until this is resolved (main is in failed integrated state — no merge proceeds) |
| `aborted_*` | §1 rows 12 / 16 | n/a (terminal) | preserve at `.claude/worktrees/aborted/<slug>+t<n>+<shortsha>+<reason>/` for forensics; delete branch |
| `stale` | staleness trigger fires | resume after stale-resolution OR user abort | preserve |

**Reuse policy (DECIDED)**:
- Retry within same task → **same worktree** (do NOT create new). Retry diff accumulates on top of prior diff in the same branch.
- Resume from `blocked` → **same worktree** if git hash matches checkpoint; if mismatch → fail-closed (PRD §1.7 step 5), prompt user to either reconcile or abort.
- Resume from `stale` → **same worktree** but staleness-trigger source must be re-validated; if base branch advanced, rebase (or block on conflict).

**Verification worktree lifecycle (DECIDED Y4 — round-3 verdict #2 fix)**:
- Created at gate-8 start (post `merge_applied`) from integration target; path: `.claude/worktrees/verify/<run_id>+t<n>+<post_merge_sha>/`
- `verification_worktree_id` + `verification_worktree_path` recorded in `post_merge_verification_started` event
- On gate-8 PASS → cleanup at task-complete step 9a.5
- On gate-8 FAIL → preserve at `.claude/worktrees/verify/aborted/<run_id>+t<n>+<post_merge_sha>+failed/` for forensics; user can inspect before choosing revert/fix path
- **Orphan recovery**: next-startup orchestrator scans `.claude/worktrees/verify/` for live verification worktrees with no matching `post_merge_verification_completed` event AND dead orchestrator pid → block with `block_type=verification_worktree_orphaned`
- **Suite policy** (S3 fix — `post_merge_skip: true` rules):
  - Default: full declared `acceptance_criteria` (affected by task) + final regression smoke
  - Per-criterion `post_merge_skip: true` allowed for type ∈ {unit, integration, e2e, smoke, behavior} with required rationale field
  - **`post_merge_skip: true` is illegal for type=`regression`** unless contract has `post_merge_regression_optional: true` (additive contract field, default false; meant only for projects with prohibitively long regression suites)

**Disk hygiene (DECIDED, round-3-fix S6 dual-base)**:
- Successful merge: worktree + branch deleted within same orchestrator run AFTER both `task_completed` and gate 8 pass.
- `aborted_*`: preserved indefinitely; user runs `flow purge-aborted <slug>` to clean.
- `blocked_post_merge`: preserved until user chooses revert/fix/split/abort. Merge stays intact in main pending decision.
- Crash recovery: orchestrator-pid file in worktree; on next orchestrator startup, orphan worktrees (no live pid + no progress.md status) → prompt user. Auto-prepare lock detection (R10) precedes orphan-worktree handling.
- **Dual base commit recording (S6)**: each worktree records both `original_base_commit` (creation) and `current_base_commit` (after rebase). After stale-rebase, shortsha in branch name still reflects ORIGINAL — disambiguate via these two commit fields, not the branch name alone.

**DECIDED (round 3, 2026-05-06, codex-refined)**:
- `Q4.1` → **DECIDED `<slug>+t<n>+<shortsha>`** (codex-refined from Claude's `<slug>+t<n>`). The shortsha is from the base commit (`git rev-parse --short HEAD`) at worktree-create time — stable for worktree lifetime; retries reuse same name. Prevents collisions when same task index reruns or restarts after archival. Branch path becomes verbose (~40 chars) but eliminates a class of resume/archive bugs.
- `Q4.2` → **DECIDED local-merge cleanup AFTER gate 8 pass** (Y2 fix — was "on local merge" pre-R3/R4; correct timing is "after local merge + post-merge verification pass"). Worktree + branch deleted only after BOTH gate 7 (local merge applied) AND gate 8 (post-merge verify) succeed AND `task_completed` event written. If gate 8 fails → `blocked_post_merge` → worktree preserved until user choice. Flow is local-first; remote PR lifecycle does not control local worktree hygiene.

## §5 Notification policy

**Question**: when does each tier fire, with what content, throttled how, and what defines "resume"?

**Per-tier firing rules (DECIDED)**:

| Tier | Firing condition | Throttle | Disabled when |
|------|------------------|----------|---------------|
| 1 (`blocked.md`) | every `block` action; on resume, moved to `archive/blocked/<ts>.md` and removed as live signal (DECIDED Q5.3) | none (always write — `notification.throttle_min` does NOT affect Tier 1) | never |
| 2 (OSC 9 + BEL) | every `block` AND every `aborted_*` final | `notification.throttle_min` (default 5 min, **0 = no throttle, every event fires**, R9 clarified); separate `notification.tier2_enabled` (default true) for full disable | terminal does not support OSC 9 (auto-detect via $TERM_PROGRAM); BEL still fires as floor (DECIDED Q5.2); `tier2_enabled=false` disables BOTH OSC 9 and BEL — only Tier 1 left |
| 3 (shell hook) | DEFERRED v0.8.2 (per PRD §3); v0.8.1 schema accepts `notification.command` field but ignores at runtime | n/a | always (v0.8.1) |

**AFK interaction (DECIDED)**:
- AFK timeout fires → check `afk_on_timeout`:
  - `wait` → row §1.11 (silent, no Tier 2 re-fire)
  - `abort` → row §1.12 (Tier 1+2 fire as final notification)
- If notification was throttled when block fired (within 5 min suppression), AFK-abort still fires Tier 2 unconditionally (it's a terminal event, not a recurring block).

**Resume contract (DECIDED)**:
- `blocked.md` MUST contain a `safe_resume_command` field, e.g.: `flow orchestrator --resume <slug> --from <checkpoint-ts>`.
- Resume reads checkpoint, replays state per PRD §1.7. Worktree git hash mismatch → fail-closed.
- Resume DOES NOT auto-re-fire any Tier 2 notification on successful resume; it logs to `decisions.jsonl` only.
- On resume, `blocked.md` is **archived to `archive/blocked/<ts>.md`** (NOT deleted; DECIDED Q5.3). Live `blocked.md` removed only after archive write succeeds (atomic move). `flow audit <slug>` reads archive for forensic replay.

**DECIDED (round 3, 2026-05-06)**:
- `Q5.1` → **DECIDED YES** (round-3-fix R9 clarified). Add `notification.throttle_min` to contract (default 5). **Semantics: `0 = no throttle, every block fires Tier 2`** (BEL safety floor still always fires regardless of throttle). For full Tier 2 disable, add separate `notification.tier2_enabled: false` field (default true). This separates "fire frequency" from "fire enable". Tier 1 always writes regardless of either field. Forward-compat: missing fields → infer defaults (`throttle_min=5`, `tier2_enabled=true`).
- `Q5.2` → **DECIDED OSC 9 + BEL only** (PRD §1.6 original). No OSC 777. BEL is the safety floor; fires regardless of OSC 9 terminal support.
- `Q5.3` → **DECIDED archive to `archive/blocked/<ts>.md`** (codex-refined path: subdirectory not sibling file, avoids `blocked.md` vs `blocked.md.*.archived` glob/parser ambiguity). Live `blocked.md` is sole authoritative signal; archive is forensic.

## §6 State ownership during partial runs

**Question**: which file is authoritative when state is mid-transition? What's the atomic unit?

**Authority hierarchy (DECIDED, per PRD §1.5 conflict resolution)**:
1. `contract.json` (versioned, atomic)
2. `decisions.jsonl` tail
3. `checkpoints/<latest>.md` + git hash anchor
4. `progress.md` (status pointer only)
5. subagent worktree journal (advisory, never authoritative)

**Atomic transitions (DECIDED)**:

| Transition | Atomic write order | Reader policy |
|------------|--------------------|--------------|
| **task complete (R3 9-step transactional sequence; Y5 explicit gap-by-gap)** | 1) gates 1–6 pass; 2) `decisions.jsonl` append `task_ready_to_merge`; 3) `checkpoints/<ts>.md` write (`phase=pre_merge`, includes `worktree_id`, `worktree_path`, `original_base_commit`, `current_base_commit`, `lifecycle_state`, `diff_hash`, `target_commit_pre_merge`); 4) `progress.md` task status → `merging`; 5) `decisions.jsonl` append `merge_started`; 6) git merge into integration target; 7) `decisions.jsonl` append `merge_applied` with `target_commit_post_merge`; 8) gate 8 in ephemeral verification worktree; 9a-PASS path: 9a.1 `decisions.jsonl` append `task_completed`; 9a.2 `checkpoints/<ts>.md` write (`phase=post_merge`); 9a.3 `progress.md` → `merged`; 9a.4 cleanup task worktree + branch; 9a.5 cleanup verification worktree; 9b-FAIL path: 9b.1 `decisions.jsonl` append `post_merge_verify_failed`; 9b.2 `blocked.md` write (`block_type=post_merge_verify_failed`); 9b.3 `progress.md` → `blocked_post_merge` (merge stays intact); 9b.4 verification worktree preserved for forensics. **Crash behavior per gap**: between any two ordered writes above → on next-startup, reader detects mismatch (e.g., `task_completed` exists but `progress.md` not `merged` → orchestrator finishes step 9a.3+; `merge_applied` without `task_completed` AND without `post_merge_verify_failed` → mid-gate-8 crash → block per R10 path). | reader picks max-ts checkpoint anchored to existing git commit; if `merge_started` exists without `task_completed` or `post_merge_verify_failed` → mid-transaction crash → block (R10 path); reader never treats `merging` as terminal |
| task block | 1) `blocked.md` write (atomic), 2) `decisions.jsonl` append, 3) `progress.md` status to `blocked` (or specific variant per §1) | reader sees `blocked.md` first as freshest signal; reads `block_type` to dispatch user-choice handler |
| task abort | 1) `decisions.jsonl` append, 2) `progress.md` status to `aborted_*`, 3) worktree move to `aborted/` (NOT deletion), 4) Tier 2 + final aborted-record fire (Tier 1 is `blocked.md`, which abort does NOT write — abort's forensic record is the `aborted/<slug>+t<n>+<shortsha>+<reason>/` directory, not a transient `blocked.md`. §1 rows 12/16 "Tier 1+2 (final)" wording means "both notification channels fire one final emission"; the on-disk aborted/ directory plays the durable role that `blocked.md` plays for `block`). | aborted state is terminal — readers should not attempt resume |
| review issue write | append `review-issues.jsonl` (single line atomic) with **canonical `issue_id` (S7)** computed as `sha256(file_path + line_range + issue_class + issue_message_normalized)[:12]` so the same issue across codex rounds collides on id (enables churn detection) | reader filters by disposition; churn = same `issue_id` recurring across N rounds |
| crash mid-transition (pre-`auto_prepare.lock`) | fail-closed → interactive (legal — user never opted in to auto for this run) | warning: "incomplete state, manual review needed"; user starts fresh in interactive |
| crash mid-transition (`auto_prepare.lock` exists, no `auto_engaged`) — DECIDED R10 | next-startup detects lock without `auto_engaged` for same `run_id/task_id` and dead pid → **`block` with `block_type=auto_prepare_interrupted`** (`block_review_state`). Choices: `resume_auto_from_prepare`, `abort_task`, `switch_to_interactive`. NOT `resume_anyway` (vague) | reader checks lock file before any other recovery action |
| crash mid-transition (post-`auto_engaged`) — DECIDED Q7.2 + §6/§7 fix | next-startup detects `auto_engaged` event without matching `task_completed`/`abort` → **`block` + user choice**: `resume_from_last_safe_state`, `abort_task`, `switch_to_interactive` | warning: "auto run interrupted; explicit user choice required" |

- `Q6.1` → **DECIDED resume from #N (last completed + 1) with safety gate, via `acceptance-progress.jsonl`** (codex-refined: log `started` + `completed` events, not just completion).

  Schema (round-3-fix R5+S2+S3 — split type/method, add attempt fields, add command refs; Y7 add criterion_hash): `{task_id, run_id, worktree_id, attempt_id, retry_idx, criterion_id, criterion_idx, criterion_hash (sha256 over normalized criterion object — for audit, distinct from command_hash), type ∈ {unit, integration, e2e, smoke, behavior, regression}, method ∈ {cmd, file_exists, json_query, http}, idempotent ∈ {true, false, unknown}, event ∈ {started, completed, timeout}, started_at, completed_at, timeout_sec, status ∈ {pass, fail, inconclusive, interrupted, timed_out}, exit_code (cmd/http only), duration_ms, stdout_log_path, stderr_log_path, command_hash}`.   **`type` is the semantic class** (PRD §1.3 acceptance types); **`method` is the execution mechanism** (the executor binding). They are orthogonal — e.g., a `behavior` type can be checked via `cmd` or `http` method. R5 fix.

  Resume policy:
  1. Read tail; find last `completed`. Resume at next idx.
  2. If a `started` exists with no matching `completed`/`timeout` → criterion was in-flight when crash hit.
  3. **In-flight criterion idempotency classification (DECIDED, round-3-fix R8 hardened)** — auto-rerun vs block:
     | Method | Default idempotency | Override path | In-flight policy |
     |---|---|---|---|
     | `file_exists` | idempotent (read-only) | n/a | safe — auto-rerun |
     | `json_query` | idempotent (read-only) | n/a | safe — auto-rerun |
     | `cmd` | **non-idempotent** (R8 hardened) | (a) command binary in `idempotent_cmd_allowlist` (default: `pytest`, `mypy`, `eslint`, `tsc`, `cargo check`, `go test`, `flow doctor`, `flow contract --validate`); OR (b) per-criterion `idempotent: {value: true, rationale: <required>, timeout_sec: <required>, side_effect_class: <enum: pure|read_only|cache_only|reversible>}` | block unless overridden via (a) or (b) |
     | `http` | GET → idempotent; POST/PUT/PATCH → non-idempotent (RFC 7231); DELETE → non-idempotent (cumulative state changes) | per-criterion `idempotent: {...}` (same shape as cmd) | by HTTP method default; user override via field |
     | `e2e` | **always non-idempotent** | **NO override accepted** (PRD §1.3 e2e fail must escalate; aligns with §1 row 6) | always block on in-flight interrupt |

  **Criterion timeout (DECIDED, round-3-fix R7 + Y1 e2e route)**: every criterion has a `timeout_sec`. Default by method: `file_exists`/`json_query` = 30s, `cmd` = 600s, `http` = 60s, `e2e` = 1800s (30 min). Per-criterion override via `timeout_sec` field. **Timeout routing depends on `type`**:
  - **`e2e` type timeout** → status `timed_out` + `inconclusive` → **§1 row 6 `blocked_escalate`** (Y1 fix — preserves "e2e never local retry" rule). User choices restricted to `{abort_task, switch_to_interactive, split_followup}`. NO `extend_timeout_and_retry` (would violate Phase 3 e2e escalation).
  - **All other types (unit/integration/behavior/smoke/regression) timeout** → status `timed_out` + `inconclusive` → **§1 row 5 block** (treated as criterion fail). User choices: `extend_timeout_and_retry`, `mark_failed_and_escalate`, `abort_task`.

  Block message names the in-flight criterion + idempotency reason + timeout reason; user picks `mark_idempotent_and_rerun`, `mark_failed_and_escalate`, `abort_task`.

- `Q6.2` → **DECIDED `supersedes: [ids]` LIST** (codex-refined from Claude's scalar). Set ONLY on retry-success after prior RED/failure. Field value = list of prior failed decision IDs that this new decision resolves (one retry can resolve multiple). NOT set on: user direction change, `rejected_with_rationale`, normal progress logs. Pairs with `review-issues.jsonl` dispositions: review issues get `disposition: fixed`; corresponding decision records get `supersedes: [...]`.

## §7 No-silent-mode-switch rule

**Question**: every code path that could degrade `auto` → `interactive` must instead block with explicit user choice. Where are the candidate degrade points, and how is each handled?

**Candidate degrade points (DECIDED audit)**:

| Site | Currently in PRD (v0.8.1)? | DECIDED behavior |
|------|---------------------------|------------------|
| missing `contract.json` | yes — falls back to interactive | KEEP fallback (this is start-of-task, not mid-task; user never opted in to auto). Not a violation. |
| invalid `contract.json` (parse fail) | yes — fail-closed → interactive | KEEP fallback, but log loud warning to `decisions.jsonl`. Start-of-task only. |
| schema version too new | yes — fail-closed → interactive | KEEP fallback. **Round-3-fix R11**: v0.8.1 reader (`scripts/flow_orchestrator.py`) MUST enforce `contract_schema_version <= CONTRACT_SCHEMA_VERSION_KNOWN_TO_RUNTIME`; v0.8.0 currently uses bare `parse_contract()` without this check. ## §8 New file schemas (round-3-fix consolidated)

### 8.1 `auto_prepare.lock` (R10)

**Purpose**: pre-`auto_engaged` boundary marker. Atomic write when contract is parsed with `autonomy_mode=auto`, BEFORE `auto_engaged` event. Deleted/renamed to `auto_prepare.consumed` after `auto_engaged` succeeds.

**Path**: `.flow/tasks/<slug>/auto_prepare.lock` (single file; NEVER live alongside an `auto_engaged` for the same `run_id/task_id`).

**Schema (JSON)**:
```json
{
  "lock_version": 1,
  "slug": "<task-slug>",
  "run_id": "<uuid>",
  "task_id": "<task-id>",
  "contract_path": "<absolute path>",
  "contract_hash": "<sha256 of contract bytes at parse time>",
  "contract_schema_version": <int>,
  "created_at": "<ISO 8601>",
  "pid": <orchestrator pid>,
  "host": "<hostname>",
  "cwd": "<absolute path>",
  "target_branch": "<integration target>",
  "intended_first_task_dispatch_at": "<ISO 8601 prediction>"
}
```

**Detection state machine (next-startup)**:

| Lock present? | `auto_engaged` for same `run_id/task_id`? | pid alive? | Action |
|---|---|---|---|
| yes | yes | n/a | normal — lock should have been consumed; orphan lock → consume + log warning |
| yes | no | dead | `block` with `block_type=auto_prepare_interrupted` (DECIDED R10) |
| yes | no | alive | active run; do not interfere |
| no | yes | n/a | normal post-engagement state |
| no | no | n/a | no auto run started; clean state |
| yes | n/a | n/a, contract changed (`contract_hash` mismatch with current contract) | `block` with `block_type=auto_prepare_interrupted`, NOT silent interactive fallback |
