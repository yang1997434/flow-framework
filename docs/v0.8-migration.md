# v0.8 Migration Guide

v0.8 introduces **autonomous mode** for Flow tasks. v0.8.0 is the foundation
ship: the `contract.json` schema is stable, dry-run preview works, and the
state-writer skeletons are in place. Autonomous *execution* unlocks in v0.8.1.

## Backwards compatibility

v0.6 / v0.7 plans run unchanged. None of the new fields are required. All
missing fields fall back to the v0.7 interactive behavior.

## What changes for you

Nothing, until you opt in by writing a `contract.json`.

## Opting in

Inside any task directory `.flow/tasks/<slug>/`:

```bash
flow contract --init <slug>      # Generates a template contract.json
$EDITOR .flow/tasks/<slug>/contract.json
flow contract --validate <slug>  # Confirm it parses
flow orchestrator --dry-run <slug>  # Preview the plan + manifest
```

Add four pointer fields to `progress.md` frontmatter:

```yaml
contract_path: contract.json
contract_schema_version: 1
autonomy_mode: auto
last_checkpoint: <iso8601>
```

## v0.8.0 limitation

`autonomy_mode: auto` is **accepted by the parser but the orchestrator
refuses to dispatch**. You'll see:

```
ERROR: v0.8.0 does not support autonomous dispatch. ... ships in v0.8.1.
```

This is intentional. The v0.8.1 release adds the safety stack (worktree-
per-task isolation, acceptance gate, notification, budget enforcement) that
makes autonomy safe to ship. Until then, the dry-run plan is your view of
what v0.8.1 *would* do.

## Forward compat

Unknown fields in `contract.json` are accepted with a warning (so a v0.8.0
parser will not crash on a contract written by v0.8.4). Invalid values for
*known* fields fail closed → contract is treated as missing → task falls
back to interactive.

`contract_schema_version > 1` fails closed (upgrade Flow before reading).
