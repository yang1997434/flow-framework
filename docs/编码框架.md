# AI 编码框架（Flow）

> **状态**：v0.2 草案。已跑过 1 个 dogfood 任务实测（建 vault `infra/` 注册表）+ 4 项行业调研锚定。**仍未在真实开发项目跑过**。
> **配套**：[[调研方法论]]、[[Trellis调研]]、[[框架对比]]
> **灵感**：Trellis 的"sub-agent 隔离 + 文件持久化"+ 现有工具组合（superpowers / planning-with-files / gstack / impeccable / vault / Codex）+ 行业 harness engineering 调研

## 7 个目标

1. **不污染主 context** — sub-agent 隔离 + 文件持久化（context firewall）
2. **多线并行** — git worktree + 每个 worktree 一个 sub-agent（按任务类型分流）
3. **历史记忆不丢** — 4 阶段强制 checkpoint + 3 tier promotion + staleness check
4. **跨模型异源审查** — Phase 3 Codex (GPT-5.5) review
5. **Token 经济** — 任务按复杂度路由模型
6. **远程 SSH 友好** — 无 GUI 依赖、相对路径、git 同步
7. **凭据安全 + 踩坑预防** — vault 不存凭据 + pitfalls 系统作为一等公民

## 文件结构

```
<project>/.flow/
  ├ config.yaml           ← 框架配置（committed）
  ├ config.local.yaml     ← 凭据 / 远程 host（gitignored）
  │
  ├ tasks/<MM-DD-slug>/   ← Lv0 task scratch（部分 committed）
  │   ├ prd.md
  │   ├ progress.md       ← 4 段落硬结构，append-only
  │   └ research/*.md     ← 仅当 Phase 1 调研时
  │
  ├ tasks/archive/<YYYY-MM>/  ← 完成任务归档（committed）
  │
  ├ ADRs/                 ← Lv1 项目级决策（committed）
  ├ patterns/             ← Lv1 项目级可复用模式（committed）
  │
  └ workspace/<dev>/      ← 个人 journal（默认 gitignored，team mode 改 committed）
       └ journal.md
```

vault 与 rules 不在项目内，是**跨项目层**：

```
~/data/knowledge-base/
  ├ patterns/<name>.md    ← Lv2 个人跨项目软规则
  ├ pitfalls/<slug>.md    ← Lv2 跨项目踩坑（v0.2 加）
  └ infra/                ← v0.2 加：基础设施注册表
      ├ servers/<name>.md
      ├ topologies/<name>.md
      └ services/<name>.md

~/.claude/rules/<topic>.md ← Lv3 个人跨项目硬规则
```

## 🔒 凭据规约（核心规则，不可绕过）

> v0.2 加。来源：dogfood 任务实测发现 6 条 auto-memory 全混着真密码（ccs root pwd / houston pwd / tinyproxy auth / target SSH key path / OPENAI_API_KEY 历史记录）。**vault 是 git-tracked + 可同步多机**，凭据写进去就泄露。

### 哪里**绝不**接受凭据

| 路径 | 为什么不能 |
|------|----------|
| vault `~/data/knowledge-base/**` | git tracked + 可能 mirror 同步到多机 |
| `.flow/` 内 committed 文件（prd.md / progress.md / ADRs/ / patterns/ / pitfalls/）| 进 git → push 到 GitHub |
| auto-memory（视情况）| 跟 Claude project 走，可能跨机器同步 |

### 哪里**可以**存凭据

| 路径 | 用途 |
|------|------|
| `~/.flow/credentials.local`（chmod 600 + gitignored）| server 密码 / API token / 第三方密钥 |
| `~/.ssh/<key>` | SSH 私钥（per-machine） |
| 环境变量 / shell rc | 短期 / 当前 session |
| 加密 secrets manager（pass / 1Password CLI）| 长期 + 跨机 |

### 引用方式

vault / `.flow/` 文件提到凭据时，用 `credentials_ref:` 字段：

```yaml
credentials_ref: ~/.flow/credentials.local#section:houston
# 或
credentials_ref: ~/.claude/projects/-data-Claude/memory/houston_proxy.md
```

**只引用位置，不复制内容**。

### Commit 前自检（建议做成 pre-commit hook）

```bash
grep -rE "(password|secret|api[_-]?key|token).*[:=]\s*['\"][^'\"]+['\"]" \
  .flow/ ~/data/knowledge-base/
```

匹中 → 检查是否漏入。Phase 3 Finish 必跑此自检。

## Phase 1 — Plan

**入口**：`/flow:start <task title>` 或主 session 检测到"建任务关键词"

**步骤**：

1. **Triage**（Haiku 4.5，1 句话分类）
   - 输出：`trivial` / `simple` / `moderate` / `complex`
   - `trivial` → 退出框架，主 session 直接做
   - 其它 → 创建 `<project>/.flow/tasks/<MM-DD-slug>/` + 写空白 `prd.md` + `progress.md`

2. **Brainstorm**（主 session, Opus 4.7 1M）
   - 用 `superpowers:brainstorming` skill 引导
   - 一次问一个问题，落 prd.md
   - **不在主 session inline WebFetch / WebSearch**

3. **Parallel research**（按需）
   - 触发：≥2 个独立调研主题 OR 涉及陌生第三方库 / 框架选型
   - 派 N 个 sub-agent 并行（默认 Sonnet 4.6；复杂方案对比升 Opus 4.7）
   - 每个 sub-agent 输出到 `<task>/research/<topic>.md`，**返回主 session 的只是 1 行 takeaway + 文件路径**

4. **ADR-lite**（写到 prd.md 末段）
   - Context / Decision / Consequences / Revisit triggers

5. **可选 codex consult**（高反悔成本决策）
   - 触发：ADR 标注 "irreversible / hard-to-revert"
   - 调 `/codex consult` 拿异源二次意见

**Phase 1 完成条件**：prd.md 写完、用户确认需求、（如有）research/ 有产物。

## Phase 2 — Execute

**入口**：Phase 1 结束 + 主 session 开始 plan implementation。

**Implement sub-agent 派发规则**（v0.2 修订，调研锚定）：

**Step 1 — 先判断任务类型**（来自 Cognition vs Anthropic 多 agent 之争的调和）：

| 任务类型 | 例子 | 派发 |
|---------|------|------|
| Breadth-first 只读 / 调研 | "对比 5 个库" / "搜索代码引用" | ✅ **可并行** sub-agent |
| 独立模块改动（无 contract 互锁）| 不同模块各自完整 + 接口稳定 | ✅ **可并行** sub-agent |
| **Interlocking design decisions** | 改动跨多文件且互相依赖 / 共同设计 contract | ❌ **单线主 session 整合**（Cognition 的 Flappy Bird 反例：并行 agent 各自决策导致拼起来不一致）|

**Step 2 — 任务类型判完，再按改动量派发**：

| 改动量 | 模型 | 是否 worktree | 派几个 |
|--------|------|--------------|--------|
| ≤2 文件，单模块 | Opus 4.7 | 否（共享 work tree）| 1 |
| 3-9 文件，1-2 独立模块 | Opus 4.7 | 是（独立 worktree）| 1 |
| **≥3 个独立模块**（替换 v0.1 "≥3 文件"，按 Amdahl's Law arxiv 2503.15703 模块边界更准）| Opus 4.7 | 是，每个独立 worktree | N |
| Novel 架构 | Opus 4.7 | 视情况 | 1（复杂决策）|

**Step 3 — Tool-count escape hatch**（来自 arxiv 2512.08296，β=−0.330）：

任务需要 **>10 个不同工具**时，**退化为单 agent**——多 agent 协调成本会反而吃掉 parallel 收益。

**关键规则——并行 sub-agent 必须 scope 互不重叠**：派之前主 session 在 `progress.md` 的 `## Plan` 段落写明：

```markdown
## Plan
- sub-agent A → scope: src/auth/**, modify: login.ts/logout.ts
- sub-agent B → scope: src/api/**, modify: handlers.ts only
- sub-agent C → scope: tests/auth/**, modify: 新增 auth.test.ts
```

**禁止 path 重叠**——这是用文件协议替代"sub-agent 间通信"的核心。

**Implement sub-agent 写完后**：
- 不 commit（commit 是 Phase 3）
- 在 `<task>/progress.md` 的 `## Execute Log` 段落 append 一行：时间 / agent / scope / outcome

**Check sub-agent**：
- 派一个 Sonnet 4.6 做 lint / typecheck / spec 一致性检查
- 跨 ≥3 层改动时升 Opus 4.7
- 自动 fix 简单问题、复杂问题报告回主 session

**卡住时（同一 bug 修了 3+ 次没好）**：触发 `/codex challenge` —— 异源模型常常一眼看穿。

## Phase 3 — Finish

1. **Final verify**（**fresh-context sub-agent**，Generator/Evaluator 模式，v0.2 修订）

   **派一个 Sonnet 4.6 sub-agent，fresh context**，只看 git diff + prd.md，**不见主 session 历史**。任务：
   - 检查 diff 是否满足 prd.md 的 Acceptance Criteria
   - 跑 lint / typecheck / 测试
   - 凭据 grep 自检（按"凭据规约"章节）

   **为什么 fresh context**：Anthropic 实证（[harness-design-long-running-apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)）——agents 评估自己工作时**会自夸**："agents tend to respond by confidently praising the work — even when the quality is obviously mediocre"。Fresh-context 隔离消除这个偏差。

   主 session 接受 sub-agent 的报告，**不重做 verify**，只写入 Verify Report。

2. **Codex 异源审查**（必触发条件）
   - 任务标 `--critical` / `--breaking`
   - 改动包含 DB migration / 公开 API contract 变更
   - 改动 ≥10 文件 OR ≥500 行
   
   **可选触发**：中等任务（3-10 文件），用户 `/flow:codex-review` 主动调用
   
   **跳过**：trivial / 纯文档 / 纯测试

3. **写 progress.md 的 `## Verify Report` 段落**：

```markdown
## Verify Report
- Self-check (Sonnet 4.6 / Opus 4.7): pass / [issues...]
- Cross-model review (Codex GPT-5.5): pass / [issues addressed: ...] / skipped (reason)
- Lint / typecheck / tests: pass
- Commit hash: <hash>
```

4. **Commit**
   - 主 session 起草 commit message
   - 如果有多个 worktree → 按顺序合并（branch 1 merge → verify → branch 2 merge → ...）
   - 用户 confirm 后真正 commit

## Phase 4 — Sediment

**核心信念**：人完任务的本能是"交了就跑"——所以 Sediment 必须是独立 phase，**不能折进 Finish**。

**写 progress.md 的 `## Sediment Notes` 段落**：

```markdown
## Sediment Notes
- ADR promoted: .flow/ADRs/<slug>.md（链接 + 1 行说明）
- Pattern promoted: .flow/patterns/<name>.md（链接 + 何时复用）
- vault candidate: [pending — 见下次 task 是否再用]
```

### Promotion 决策

| 触发 | 动作 |
|------|------|
| 这个决策"值得未来回看" | Lv0 → Lv1：写到 `.flow/ADRs/<slug>.md` |
| 这个 pattern 在本任务沉淀 | Lv0 → Lv1：写到 `.flow/patterns/<name>.md` |
| 同 pattern 在 ≥2 个 task **跨 ≥2 项目**用过 | Lv1 → Lv2：promote 到 vault `~/data/knowledge-base/patterns/` |
| pattern 用 ≥3 次无例外 + 人工最终确认 | Lv2 → Lv3：写到 `~/.claude/rules/<topic>.md` |

### 容量上限（强制 curation）

| 层 | 单文件上限 | 超限处理 |
|----|----------|---------|
| `.flow/ADRs/<slug>.md` | <500 行 | 拆按主题分文件 |
| `.flow/patterns/<name>.md` | <300 行 | 拆 / 抽离子 pattern |
| vault `patterns/<name>.md` | <300 行 | 同上 |
| `~/.claude/rules/<topic>.md` | <200 行 | 拆主题 / 降级到 vault |

### Staleness 检查

`/flow:sediment-verify`（Phase 4 末自动跑，也可单独触发）：

1. 列出本 task 引用的所有 ADR / pattern / rule
2. 对每个引用的文件路径：
   - 该路径还存在吗？
   - 该路径在最近 5 commit 内是否被改过？
3. 标 STALE 的项 → 提示用户 keep / update / remove

### Conflict 检测

`/flow:start` 时自动跑（每个新 task 开始）：

1. 列出本 task 加载的所有 rules + ADRs + patterns
2. 派一个 Sonnet 4.6 sub-agent 做 pre-flight：
   ```
   "下面是本 task 加载的 N 条规则。其中是否有互相矛盾的？
    若有，列出冲突对 + 建议哪条优先。"
   ```
3. 报告冲突给用户，**不自动消解**——用户决定

## 模型路由表

| 任务类型 | 模型 | 何时升级 |
|---------|------|---------|
| Triage | Haiku 4.5 | — |
| 主 session 编排 | **Opus 4.7 (1M)** | — |
| Research sub-agent | **Sonnet 4.6 (1M GA)** | 5+ 方案对比时升 Opus 4.7 |
| Implement sub-agent | **Opus 4.7** | — |
| Check sub-agent（同层）| Sonnet 4.6 | — |
| Check sub-agent（跨 ≥3 层）| **Opus 4.7** | — |
| Cross-model review | Codex (GPT-5.5) | — |
| Conflict pre-flight | Sonnet 4.6 | — |

总成本估算：相比"全 Opus"省 ~2-3x；相比"全 Haiku"质量提升明显。

## Codex 异源审查

| 时机 | 模式 | 触发 |
|------|------|------|
| Phase 1 末（架构决策）| `/codex consult` | ADR 标 irreversible |
| Phase 2 卡住 | `/codex challenge` | 同 bug 修 3+ 次没好 |
| Phase 3（commit 前）| `/codex review` | critical / breaking / DB migration / API change / ≥10 文件 / ≥500 行 |

不在调研类（Phase 1 research）用 codex 的理由：调研主要是 fetch + 总结，单模型够用。

## 远程 SSH 适配

3 条规约：

1. **`.flow/` 文件用相对路径**——不写绝对 `/home/yangpeng/...`
2. **凭据走 `config.local.yaml`（gitignored）或环境变量**——不进 `.flow/`
3. **不假设远端有 GUI 工具**——impeccable / browser 是可选层，框架核心不依赖

`config.local.yaml` 模板：

```yaml
# gitignored — 每台机器自己一份
machine_id: ccs-maruiao  # 或 storage-azure / target-coolify
remote_targets:
  prod-dev:
    host: 139.99.125.49
    user: yangpeng
    docker_root: /opt/projects/<this-project>
```

`.flow/.runtime/` 用 `machine_id + session_id` 做 key，避免镜像同步把另一台 active task 拉过来。

## 踩坑系统（pitfalls，v0.2 加）

行业其他工具把踩坑混在 spec / rules / feedback 里——**Flow 把它独立成一等公民**。

### 文件结构

```
<project>/.flow/pitfalls/<slug>.md       ← Lv1 项目级
~/data/knowledge-base/pitfalls/<slug>.md ← Lv2 跨项目（vault）
~/.claude/rules/pitfalls-<topic>.md      ← Lv3 硬规则
```

### Pitfall 文件 schema

```yaml
---
name: <slug>
date: <YYYY-MM-DD>
project: <name 或 "cross">
severity: low | medium | high      # 重犯代价
status: active | obsolete           # 留 history 不删
trigger_paths:                      # 自动加载的命中条件
  - <path glob 或 library 名>
last_verified: <YYYY-MM-DD>
---

## Symptom（看到什么）
## Root cause（实际原因）
## Fix（这次怎么解决的）
## Prevention（下次怎么避免）
## Why it matters（重犯代价）
## References
```

### 自动加载机制（trigger_paths）

Phase 1 任务启动时：
1. 扫 prd.md 提到的文件路径 / library 名 / 命令
2. 匹配各 pitfall 的 `trigger_paths` glob
3. 命中的自动加载到主 session context（按相关性排序，限 top-N）
4. **不全部加载**——避免 context 噪音

### 捕获时机（4 个，按侵入度排）

| 触发 | 谁触发 | 侵入度 |
|------|-------|-------|
| **Phase 4 Sediment 询问** | 框架默认 | 低（一句"本任务有踩坑值得记吗？y/n/skip"）|
| **Debug loop 检测**（同 bug 修 3+ 次）| 框架自动 | 中（强制提示捕获）|
| **`/flow:pitfall <symptom>`** | 用户主动 | 零 |
| **Sub-agent 报错时** | 主 session | 中（追问"已知坑还是新坑？"）|

**反骚扰**：Phase 4 询问连续被 skip 3 次 → 自动降低询问频率（每周一次）。

### 容量上限

- 单 pitfall body <800 字符（Letta-anchored）
- trigger_paths 段 <300 字符
- 超限触发拆分或归档（status: obsolete）

### 与 Claude Code auto-memory `feedback` 类型分工

| auto-memory `feedback` | 框架 `pitfalls/` |
|----------------------|----------------|
| 关于"Claude 怎么和这个用户工作" | 关于"这个项目 / 技术栈本身有什么坑" |
| 跨所有项目随用户走 | 项目级 / 跨项目库级 / 硬规则级 |
| 用户偏好驱动 | 事故驱动 |
| 例子：用户喜欢简短回复 | 例子：[[catbus-server-argparse-host]] argparse default 覆盖函数 default |

两者**不冲突，互补**。

## 与现有工具的边界（编排 vs 执行）

**框架是编排层，不是执行层**——它告诉你"什么时候用什么"，不重复造工具。

| 框架职责 | 现有工具 |
|---------|---------|
| Phase 切换 / 文件落点 / 任务类型分流 | — |
| 量化阈值（派几个 sub-agent / 何时升模型）| — |
| ADR-lite + pitfall + 凭据规约模板 | — |
| Promotion 阶梯（Lv0→Lv3）| — |
| Staleness + Conflict 自动检测 | — |
| **行为执行** | superpowers / impeccable / gstack / pr-review-toolkit / document-skills / planning-with-files / actionbook / obsidian / baoyu-skills 等 |
| **凭据存储** | `~/.flow/credentials.local` / `~/.ssh/` / 环境变量 |
| **Token 路由** | Agent 工具的 `model` 参数 |

## 反模式清单（v0.2 加，调研锚定）

行业广泛劝避的反模式 + Flow 怎么避免：

| 反模式 | 出处 | Flow 怎么避免 |
|-------|------|------------|
| Kitchen-sink session（混任务）| Claude Code Best Practices | 1 task 1 directory，新 task `/flow:start` |
| 修了又错、错了又修 | Claude Code | 3+ 次失败 → `/codex challenge` / 后备 `/clear` |
| 过长 CLAUDE.md（重要规则被淹没）| Claude Code | "removing this would cause mistakes" 测试，否则删 |
| 信而不验（trust-then-verify gap）| Claude Code | Phase 3 fresh-context sub-agent verify required |
| 无界探索（"investigate X" 读 100 文件）| Claude Code | 调研走 sub-agent + scope |
| 塞太多 MCP（context 爆）| HumanLayer | 优先 CLI > MCP（well-trained 域）|
| Role-specialised sub-agent ("frontend engineer")| HumanLayer | Flow 用**功能**隔离（research/implement/check）非角色 |
| Self-review 当质量门 | Anthropic | Generator/Evaluator: fresh-context verify |
| 假设 long context 解决问题 | HumanLayer | "performance degrades as context fills" |
| 多 agent 跑互相依赖的设计决策 | Cognition | 任务类型判别（Phase 2 Step 1）|
| 提前 mark 完成不验证 | Anthropic | progress.md verify report 必填 |
| Scaffolding accretion（积累不必要的 harness 逻辑）| Anthropic | 周期性"删 harness 逻辑"——模型升级了哪些 scaffolding 已经不需要 |
| Auto-generated AGENTS.md / agentfiles | ETH Zurich 实证（20%+ 成本，性能反降）| 手写或半自动，**不**让 LLM 全自动生成 |
| Vault / committed 文件存凭据 | Flow dogfood 实测 | 凭据规约 + commit 前 grep 自检 |

完整行业引用见 [[框架对比]] 及调研产物 `/tmp/flow-research/`。

## 怎么用（User Runbook，v0.2.1 加）

> 之前章节都是"什么是什么"。这一节是"我手上有任务，下一步具体做什么"。

### Day 1：让框架在一个项目里能跑起来

```bash
# 1. 进任意项目根目录（或在新项目里）
cd <project-root>

# 2. 建 .flow/ 骨架（手动，slash command 还没实现）
mkdir -p .flow/{tasks,ADRs,patterns,pitfalls,workspace/$USER,archive}

# 3. 写 .gitignore（项目根追加）
cat >> .gitignore <<'EOF'
.flow/.runtime/
.flow/config.local.yaml
.flow/workspace/$USER/
EOF

# 4. （可选）凭据存哪
mkdir -p ~/.flow && chmod 700 ~/.flow
touch ~/.flow/credentials.local && chmod 600 ~/.flow/credentials.local
```

之后**每个新任务**都按下面"任务流程"走。

### 任务流程（典型任务一遍走完）

```
你给 Claude 一句任务描述
        ↓
[1] Triage（Haiku 4.5 判 trivial/simple/moderate/complex）
        ├─ trivial → 退出框架，直接做
        └─ simple+ → 创建 .flow/tasks/<MM-DD-slug>/{prd.md, progress.md}
                     ↓
[2] Phase 1 — Plan
    主 session 调 superpowers:brainstorming
        ↓
    （UI 任务）追加 impeccable:shape
        ↓
    （需调研）派 N 个 research sub-agent → research/*.md
        ↓
    收敛 prd.md + ADR-lite
        ↓
    （高反悔决策）调 gstack:codex consult
        ↓
[3] Phase 2 — Execute
    Step 1: 任务类型判别（Interlocking → 单线 / Independent → 可并行）
    Step 2: 改动量 + 模块判派几个 sub-agent
    Step 3: tool-count escape hatch（>10 工具退化单 agent）
        ↓
    主 session 在 progress.md ## Plan 段写 sub-agent scope 划分
        ↓
    派 implement sub-agent(s)（Opus，worktree=按规则）
    （UI 任务）调 impeccable:frontend-design
    每个 sub-agent 完成 → progress.md ## Execute Log append
        ↓
    派 check sub-agent（Sonnet 4.6）
    （卡 3+ 次）调 gstack:codex challenge
        ↓
[4] Phase 3 — Finish
    派 fresh-context sub-agent（Sonnet）只看 diff + prd 做 verify
    跑 lint / typecheck / 测试 / 凭据 grep 自检
        ↓
    （触发条件）调 gstack:codex review（必/可选/跳）
    （UI）调 impeccable:audit + polish + gstack:design-review
        ↓
    progress.md ## Verify Report 段写完
        ↓
    主 session 起草 commit message → 用户 confirm → commit
        ↓
[5] Phase 4 — Sediment（独立 phase，不能跳）
    progress.md ## Sediment Notes 段
        ↓
    决定 promote：
      - ADR / pattern → .flow/ADRs/ 或 .flow/patterns/
      - 跨项目用过的 pattern → vault patterns/
      - 用 ≥3 次无例外 → ~/.claude/rules/
      - 踩坑（任意触发）→ pitfalls/
        ↓
    调 yangpeng-claude-skills:save 写 session 断点
    （周末）调 gstack:retro 周回顾
        ↓
    任务归档到 .flow/tasks/archive/<YYYY-MM>/
```

### 三个具体场景走一遍

#### 场景 A：小后端 bug fix（complexity: simple）

```text
你："API /dispatch 大于 1MB 的 body 就 400 — 修一下"

Claude:
  Phase 1: Triage = simple → 建 .flow/tasks/05-04-fix-dispatch-large-body/
  Phase 1: brainstorming → prd.md（Goal: 找 1MB 上限源 + 修；≤2 文件 + 单模块）
  Phase 1: 无需调研、无需 ADR
  Phase 2: 任务类型 = 独立模块 / 改动量 = ≤2 文件 → 1 个 implement sub-agent，无 worktree
  Phase 2: TDD 调用 → 先写复现测试
  Phase 2: implement sub-agent 找到 bodyParser size 默认 100kb，patch
  Phase 2: check sub-agent 跑测试 ✅
  Phase 3: fresh-context verify sub-agent → pass
  Phase 3: 改动 = 2 文件 < 10 文件 → Codex review **跳过**（按规则）
  Phase 3: code-review:code-review → 小 PR 默认
  Phase 3: commit
  Phase 4: 决定：本任务的 bug 是 express bodyParser 默认配置坑
    → promote 到 vault pitfalls/express-bodyparser-size-limit.md（Lv2，跨项目通用坑）
  Phase 4: save 断点
```

#### 场景 B：UI 新页面（complexity: moderate）

```text
你："催巴 web 加用户设置页（profile / 头像 / 偏好 / 通知）"

Claude:
  Phase 1: Triage = moderate / 任务类型 = 前端 UI
  Phase 1: brainstorming + impeccable:shape（UX 简报：4 sections + edit/save 模式）
  Phase 1: 派 research sub-agent 扫现有 design token + 类似页面
  Phase 1: prd.md 收敛 + Acceptance Criteria（4 sections present + edit save works）
  Phase 2: 任务类型 = 独立模块 / 改动 = 3-9 文件 → 1 个 implement sub-agent + worktree
  Phase 2: implement sub-agent 调 impeccable:frontend-design 实施
  Phase 2: check sub-agent 验
  Phase 3: fresh-context verify
  Phase 3: 改动 = 5 文件 + 中等任务 → Codex review **可选**，用户决定
  Phase 3: impeccable:audit + polish
  Phase 3: gstack:design-review 真浏览器视觉审计
  Phase 3: commit
  Phase 4: 决定：抽出可复用 SettingsCard 组件
    → promote 到 .flow/patterns/settings-card.md（Lv1，本项目可复用）
  Phase 4: save 断点
```

#### 场景 C：跨层重构（complexity: complex）

```text
你："把 dispatch 从同步改成基于队列的异步处理（DB / 队列 / API / 前端 4 层都要动）"

Claude:
  Phase 1: Triage = complex / 任务类型 = 跨层（interlocking design）
  Phase 1: brainstorming
  Phase 1: 派 3 个 research sub-agent 并行（队列方案对比 / 现有 dispatch 调用图 / 兼容性）
  Phase 1: 收敛 prd.md + ADR-lite（reverible: hard / 触发 codex consult）
  Phase 1: gstack:codex consult → GPT-5.5 二次意见
  Phase 2: 任务类型 = **interlocking design decisions** → **单线主 session 整合**（不并行）
  Phase 2: 主 session 自己 implement，但派 sub-agent 处理独立子模块（如队列 client）
  Phase 2: 多个 commit，每个 commit 后 check sub-agent
  Phase 3: fresh-context verify
  Phase 3: 改动 = 15 文件 + DB migration + API 变更 → Codex review **必触发**
  Phase 3: code-review + pr-review-toolkit:review-pr（大 PR）
  Phase 3: commit
  Phase 4: 决定：写 ADR `.flow/ADRs/dispatch-async-migration.md`
  Phase 4: 决定：抽 queue-pattern 到 .flow/patterns/
  Phase 4: 周末 gstack:retro 总结
  Phase 4: save 断点
```

### 反模式（"我不该做什么"）

- ❌ 一次给 Claude 5 个不相关任务（kitchen-sink）
  - ✅ 1 task 1 directory，新 task 重新 `/flow:start`

- ❌ 卡 5 次仍然让 Claude 自己换思路修
  - ✅ 3+ 次失败 → `gstack:codex challenge` 或 `/clear`

- ❌ 写 200 行的 prd.md 把所有可能想到的都列上
  - ✅ Acceptance Criteria + Out of Scope 互相约束，prd.md 50-150 行为佳

- ❌ Phase 4 Sediment 心想"这次没什么 sediment 的" 然后跳过
  - ✅ 哪怕只写一行"本任务无新 ADR / pattern"也得写——强制让你思考

- ❌ 凭据写进 prd.md / progress.md 图方便
  - ✅ `credentials_ref:` 引用，凭据走 `~/.flow/credentials.local`

完整反模式见上文 [反模式清单](#反模式清单).

## 渐进采用路径

不必一次全用。建议三阶段：

### 阶段 A：核心流程（最低成本上手）
- 4 phase 命名 + 4 段落 progress.md
- prd.md / research/ 文件协议
- Phase 4 sediment 强制为独立 phase（即使只是写一行"无新东西"）

### 阶段 B：Sub-agent 派发 + 模型路由
- Implement / Check / Research sub-agent 角色
- 模型路由表
- 并行 worktree

### 阶段 C：自动化检查
- staleness check
- conflict detection pre-flight
- Codex 异源审查

每个阶段独立可用，B 不依赖 A 全部到位、C 不依赖 B 全部到位。
