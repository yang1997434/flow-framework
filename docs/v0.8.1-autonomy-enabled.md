# v0.8.1 — Autonomy Enabled (Migration Guide)

> Released: 2026-05-07.
> Audience: existing v0.8.0 users who want to enable orchestrator-driven
> task execution.

v0.8.0 shipped the contract schema and a refusing dry-run. v0.8.1 lifts the
refusal: an 8-gate safety stack now executes per-task, with worktree
isolation, manifest enforcement, codex review, acceptance criteria, atomic
merge, post-merge verify, crash recovery, and nested-autonomy abort.

This guide walks the new fields, defaults, and the operator-visible
artifacts (`blocked.md`, `acceptance-progress.jsonl`, `auto_prepare.lock`).

---

## 1. What's new

- `autonomy_mode: auto` now executes (v0.8.0 refused with exit 2).
- 8 gates per task: baseline → subagent → manifest verify → codex review →
  acceptance criteria → regression smoke → local merge → post-merge verify
  (in an ephemeral verification worktree).
- `Notifier` (3-tier): Tier 1 always writes `blocked.md`; Tier 2 emits
  OSC 9 + BEL with throttle; Tier 3 schema-only (deferred to v0.8.2).
- AFK + Budget contract fields parse but are runtime-deferred to v0.8.2.
- Crash recovery: 5-state classifier covers pre-lock, lock+dead-pid,
  post-`auto_engaged`, mid-merge, verification-orphan.
- Staleness: 5 explicit triggers wired into `flow doctor`.
- Nested-autonomy abort: `FLOW_AUTONOMY_PARENT_PID` env-var guard refuses
  re-entry at orchestrator entry.

## 2. How to enable

1. In Phase 1, write a `contract.json` under `.flow/tasks/<slug>/` with
   `autonomy_mode: "auto"` plus at least one `acceptance_criteria` entry
   and a `scope` block (`allowed` + `forbidden` globs).
2. Run `flow contract --validate <slug>` to confirm parse + cross-field
   rules pass.
3. Run `flow doctor` to confirm staleness checks are clean and no
   orphan verification worktree exists from a prior run.
4. Run `flow orchestrator --auto-execute <slug>`. The orchestrator writes
   `auto_prepare.lock`, dispatches per-task subagents into per-task
   worktrees, runs the 8 gates, and emits `decisions.jsonl` events.

If any gate fails, a `blocked.md` is written with `safe_resume_command`
and the orchestrator exits non-zero. Resume by re-running the printed
command after the user resolves the block.

## 3. `acceptance_criteria` writing guide

Each criterion has a `description`, a `type` ∈ {unit, integration, e2e,
smoke, behavior, regression}, a `method` ∈ {cmd, file_exists, json_query,
http}, plus a method-specific required field (`command` / `path` / `url`).

`type` chooses the semantic policy (e2e bypasses retry; regression checks
post-merge); `method` chooses how the runner invokes it.

```json
{
  "description": "Smoke + unit suite green",
  "type": "smoke",
  "method": "cmd",
  "command": "bash tests/smoke/run.sh",
  "timeout_sec": 600
}
```

Per-method timeout defaults: cmd=120s, file_exists=5s, json_query=15s,
http=30s. e2e always overrides to 600s. `idempotent` overrides are
forbidden on `type: e2e` (T9 always blocks in-flight e2e).

## 4. Notification

- `notification.throttle_min` (default 5): suppress duplicate
  `(task_id, issue_id)` Tier-2 notifications within N minutes.
  `0` disables throttle entirely (every event fires).
- `notification.tier2_enabled` (default true): when false, no OSC 9 + no
  BEL — Tier 1 (`blocked.md`) still writes regardless.
- Tier 1 (`blocked.md`) always fires.
- Tier 3 (`notification.command`) is schema-only in v0.8.1 and deferred
  to v0.8.2; setting it logs a stderr note but does not invoke the shell.
- Auto-detect: kitty / Ghostty / iTerm.app emit OSC 9 + BEL; unknown
  terminals fall back to BEL only.
- Resolved blocks archive to `.flow/tasks/<slug>/archive/blocked/<ts>.md`
  on next dispatch.

## 5. AFK semantics (deferred)

`afk_timeout_min` and `afk_on_timeout` parse and store but no `AFKDetector`
runs in v0.8.1. The fields are forward-compat for v0.8.2's
`AFKDetector.wait_with_24h_cap()` — a 24h hard cap with mechanical
activity signals (typing / new commits / git fetch). v0.8.1's contract is
already correctly shaped for v0.8.2, so no re-edit is required when AFK
lands.

## 6. Crash recovery walkthrough

The 5-state classifier in `flow_orchestrator.py` decides what each
`flow orchestrator --auto-execute <slug>` re-entry does:

1. **Pre-lock**: no `auto_prepare.lock` → fresh start.
2. **Lock present + dead pid**: stale lock; offer to clear via
   `blocked.md` choice.
3. **Post-`auto_engaged`**: a task's worktree exists but the dispatch
   was killed; resume from last persisted gate.
4. **Mid-merge**: a partial 9-step transactional merge is detected via
   the 9-step state file; the runner aborts with a `blocked.md` and the
   user picks rollback / manual retry / proceed.
5. **Verification-orphan**: an ephemeral verify worktree is detached from
   any task; `flow doctor` flags it; user runs `flow doctor --gc-verify`.

## 7. Reading `blocked.md`

Every block writes a `blocked.md` with frontmatter:

```yaml
---
task_id: t3
issue_id: gate6_smoke_red
tier: 1
created_at: 2026-05-07T03:14:00Z
safe_resume_command: flow orchestrator --auto-execute <slug>
auto_engaged: true
---
```

Body holds the reason + remediation hints. `safe_resume_command` is
literal — copy/paste once the issue is resolved. If `auto_engaged: false`,
recovery is via the dispatcher choice menu, not direct re-run.

## 8. `flow audit <slug>`

After a task ships (or aborts), `flow audit <slug>` replays archived
state from `archive/blocked/` and `decisions.jsonl` to print the
chronology. Useful for retrospectives — confirms what each gate decided
and which `blocked.md` choices the user made.

## 9. Migration from v0.8.0

v0.8.0 contracts continue to parse. v0.8.1 only adds fields:

- top-level: `idempotent_cmd_allowlist`, `post_merge_regression_optional`
- `notification.throttle_min`, `notification.tier2_enabled`
- `budget.max_codex_rounds_per_task`
- per-criterion: `idempotent` (object), `timeout_sec`, `post_merge_skip`

Missing fields default conservatively (see §3 + §4). v0.8.0 readers
encountering v0.8.1 contracts accept unknown fields with a warning
(`tests/smoke/test_v080_unknown_field_compat.py` enforces this).

## 11. Troubleshooting

- **OSC 9 not firing on CI**: CI terminals usually aren't kitty/Ghostty/
  iTerm; the runner falls back to BEL. Set `tier2_enabled: false` to
  silence both — Tier 1 still writes `blocked.md`.
- **Lockfile-staleness false positive**: dependency files (`uv.lock`,
  `package-lock.json`) get touched by IDEs without a real change.
  `flow doctor` reports the trigger; if the lockfile content is
  unchanged from the base commit, run `git restore <lockfile>` and
  re-validate.
- **Nested-autonomy abort**: if a task subagent itself triggers
  `flow orchestrator --auto-execute`, the env-var guard
  (`FLOW_AUTONOMY_PARENT_PID`) refuses with exit 3 — by design. Move the
  inner workflow to a separate `<slug>` and run it after the outer one
  completes.
- **`blocked.md` resume loop**: if `safe_resume_command` re-blocks
  immediately, inspect `decisions.jsonl` for the last `*_failed` event;
  the issue is upstream of the gate that re-triggered (typically a
  staleness trigger).
- **`flow doctor` complains about a verify worktree**: leftover from a
  killed run. Run `flow doctor --gc-verify` (or remove
  `.claude/worktrees/verify/<run_id>+t<n>+<sha>/` manually).
