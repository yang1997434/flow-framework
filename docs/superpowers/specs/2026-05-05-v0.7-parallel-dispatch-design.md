**Author**: yangpeng + Claude

**Approver**: yangpeng

## Problem statement

Flow Phase 2 (`execute`) currently dispatches implementer subagents **sequentially** via `superpowers:subagent-driven-development`. That skill's guidance line ("Never dispatch multiple implementation subagents in parallel — conflicts") is correct **for the single-task case**, but it forces all multi-task plans into serial execution even when tasks touch disjoint files.

v0.6.0 implementation took 7 sequential subagent waves; **5 of them were file-level disjoint** and could have run concurrently. Wall time roughly halves if those run in parallel.

v0.7 adds a **wave-based parallel dispatcher** that runs only proven-independent tasks concurrently, with conservative defaults. The controller (main Claude session) is the parallelism judge — authors write linear plans, the controller decomposes into waves at Phase 2 entry.

---

## Design philosophy

1. **Conservative by default** — any uncertainty about independence → serial. Match Bazel's hermeticity spirit, not Make's silent races.
2. **Mechanical proof primary, LLM judgment secondary** — file-write disjointness is the positive proof; LLM concept check can only veto (downgrade parallel→serial), never elevate (serial→parallel).
3. **Reproducibility over flexibility** — wave decomposition is cached and keyed by plan hash + base commit + controller model. Same inputs → same waves.
4. **Verification at every boundary** — declared `writes:` is a hypothesis; the wave integration step verifies actual `FILES_CHANGED ⊆ declared writes`. Violation → wave contaminated, stop.
5. **Backwards compat = zero regression** — plans without `writes:` field auto-fallback to all-serial. Capability missing → fallback to existing `subagent_dispatch`. v0.6.x plans run unchanged.

## Architecture

### Phase 2 orchestrator flow

```
Phase 1 (plan)
  ↓ produces implementation_plan with optional `writes:` per task

Phase 2 (execute) — orchestrator
  Step A: read plan, invoke `wave_planning` capability
    → planner reads cache (.flow/tasks/<slug>/wave-decomposition.json)
    → if plan_hash + base_commit match: reuse cached waves
    → else: compute waves via independence algorithm, write cache
    → emit waves[]
  Step B: for each wave in waves[]:
    - if wave.size == 1 → invoke `subagent_dispatch` capability (existing path)
    - if wave.size  > 1 → invoke `wave_dispatch` capability (new flow-wave-runner)
  Step C: at wave barrier, controller (main session) judges:
    - all tasks reached terminal state?
    - any failed_blocking / blocked / timed_out → block next wave; controller must explicitly waive to advance
    - critical_blocking from code-quality reviewer → must fix before advancing
    - else → advance to wave[k+1]

Phase 3 / Phase 4 — unchanged
```

### Plan schema extension

```yaml
# implementation_plan.yaml
tasks:
  - id: task-1-update-phase1-skill
    title: "Update Phase 1 SKILL.md to add Engineer hat"
    writes:                                # OPTIONAL, glob list — hard independence evidence
      - claude/skills/flow/flow-phase1-plan/SKILL.md
    reads:                                 # OPTIONAL, hint only — does not affect independence
      - docs/Skills-Phase映射.md
    description: |
      ...
```

Field semantics:
- `writes`: file globs the task plans to write. Absent → task is **strictly serial** (cannot join any wave).
- `reads`: file globs the task plans to read. **Hint only** — N tasks reading the same file does not block parallel.
- Plans without `writes:` field anywhere → entire plan runs all-serial (v0.6.x behavior preserved).

### Independence algorithm

```
input: ordered list of tasks (plan order = implicit dependency declaration)
output: list of waves (each wave is a list of tasks)

remaining = tasks  # in plan order
while remaining:
  wave = [remaining.pop(0)]
  for t in remaining[:]:
    if can_join_wave(t, wave):
      wave.append(t)
      remaining.remove(t)
      if len(wave) >= CAP:    # default 3, see §Cap
        break
  emit wave
  
can_join_wave(t, wave):
  for w in wave:
    # Step 1: mechanical disjointness (must pass)
    if t.writes is None or w.writes is None:
      return False                        # missing writes → strict serial
    if any glob in t.writes overlaps any glob in w.writes:
      return False                        # writes overlap → serial
    if any glob in t.writes ∈ SHARED_ARTIFACTS or any glob in w.writes ∈ SHARED_ARTIFACTS:
      return False                        # shared artifact → forced serial
    if any glob in t.writes contains broad-pattern (e.g., '**/*' or '**'):
      return False                        # too broad to verify → serial
    
    # Step 2: LLM concept check (veto only — can downgrade parallel → serial, never elevate)
    if controller LLM-judges "t and w conceptually coupled":
      return False                        # e.g., both modify same orchestrator behavior across files,
                                          #       or t exports a type w consumes,
                                          #       or t is docs reflecting code w writes
  return True
```

**Why mechanical-first**: codex consult flagged that LLM-only positive proof of independence is unsafe. Mechanical disjointness gives a deterministic positive answer; LLM only refines downward.

**LLM concept check signals** (controller veto on any of these):
- t introduces a new type, w consumes a similar type → contract race
- t and w both touch the same conceptual contract even in different files (e.g., capability registry semantics)
- t is documentation of behavior w implements → drift risk
- t and w both modify the orchestrator dispatch logic in different skills

### SHARED_ARTIFACTS list (initial)

Stored in `claude/skills/flow/flow-wave-runner/SHARED_ARTIFACTS.md`. Any task writing to these globs forces the entire wave to serial regardless of declared disjointness:

```
package.json, package-lock.json, *.lock
yarn.lock, pnpm-lock.yaml, Cargo.lock, go.sum, go.mod, requirements.txt, poetry.lock
.flow/skills-map.md
claude/capabilities/*.json
VERSION
CHANGELOG.md
```

Extended over time as we observe collisions in practice.

## Wave runtime (`flow-wave-runner` skill)

### Concurrency structure

```
Wave[k] = [task_1, ..., task_N]   where N ≤ CAP (default 3)

Time slot 1: 2N concurrent subagents (paired implementer + spec reviewer per task)
  paired_1: implementer_1 → spec_reviewer_1 → fix-loop (max 2 retries) → terminal
  paired_2: implementer_2 → spec_reviewer_2 → fix-loop (max 2 retries) → terminal
  ...
  paired_N: implementer_N → spec_reviewer_N → fix-loop (max 2 retries) → terminal
  
  (paired runs in parallel across N; within each paired, sequential)

Time slot 2: sequential merge into wave integration branch (controller)
  for task in wave (plan order):
    git merge worktree-of-task → wave-branch
    verify: actual FILES_CHANGED ⊆ declared task.writes  (strict subset, no exceptions)
    on violation → mark wave contaminated, abort merge, escalate
    (rationale: any undeclared write — even expected ones like formatter output
     or generated files — must be added to the task's `writes:` glob explicitly.
     Strict subset is the safest invariant.)

Time slot 3: 1 subagent
  code_quality_reviewer reads merged wave-branch diff (cross-task consistency)
  
Time slot 4: main controller (me)
  collect N status reports + 1 code-quality report
  judge: all terminal? critical issues fixed? integration coherent?
  → advance to wave[k+1] OR escalate
```

**Why sequential merge** (not octopus): codex flagged that even disjoint declared writes can collide on generated files (lockfiles, formatted output, snapshots). Sequential merge in plan order, with FILES_CHANGED verification per merge, catches contamination at the earliest task that produced it.

**Concurrency cap (CAP)**:
- Default: **3** (peak in-flight: 3 implementers + 3 spec reviewers + 1 code reviewer = 7)
- Opt-in: **4** via project config `claude/flow.config.json` → `parallel_dispatch.cap: 4` (peak: 9)
- N>CAP independent tasks → chunked into ⌈N/CAP⌉ waves of size ≤CAP

### Subagent return contracts

Each subagent must return ONLY these fields. Total budget enforced in subagent prompt.

**Implementer** (~600 token cap):
```
STATUS: done | done_with_concerns | blocked | failed_blocking | failed_nonblocking_waived | timed_out | cancelled
COMMIT_SHA: <full sha or "none">
FILES_CHANGED: [list of paths actually written]
SUMMARY: 3-5 bullets — what changed and why
CONCERNS: [] or short list
NEXT: 1-2 lines — integration notes for controller
```

**Spec reviewer** (~400 tokens):
```
VERDICT: approved | issues_found
COMMIT_SHA: <reviewed sha>
ISSUES: [] or list of {file, line, severity: critical|important|minor, fix_required: bool}
RETRY_COUNT: 0 | 1 | 2
```

**Code quality reviewer** (~600 tokens, runs once per wave at integration):
```
WAVE_VERDICT: approved | critical_blocking | minor_deferred
CRITICAL_ISSUES: []  # block next wave
MINOR_ISSUES: []      # write to .flow/tasks/<slug>/followup.md
CROSS_TASK_OBSERVATIONS: 2-3 bullets — duplication, style drift, contract drift
```

Diffs are NOT in returns. Controller fetches via `git show <sha>` only when integration check requires it.

### Terminal state taxonomy

| State | Meaning | Blocks next wave by default? | Controller can waive? |
|-------|---------|------------------------------|----------------------|
| `done` | Task complete, spec approved, no concerns | No | n/a |
| `done_with_concerns` | Task complete but concerns flagged | No (concerns deferred) | n/a |
| `blocked` | Implementer cannot proceed, no commit | **Yes** | Yes (controller can waive if no downstream task depends) |
| `failed_blocking` | Spec reviewer found unfixable issues after 2 retries; impacts later waves | **Yes** | No (must fix) |
| `failed_nonblocking_waived` | Failed but controller waived (after explicit dependency check) | No | Already waived |
| `timed_out` | Subagent exceeded wall-clock budget | **Yes** | Yes |
| `cancelled` | Manually cancelled by controller | **Yes** | Yes |

Wave barrier requires every task in `Wave[k]` to be in some terminal state. Default policy: any state in {`blocked`, `failed_blocking`, `timed_out`, `cancelled`} blocks `Wave[k+1]` until controller either fixes or explicitly waives with rationale logged to `.flow/tasks/<slug>/wave-decisions.log`.

### Failure / retry tree

```
implementer returns
  ├─ STATUS=done → spec_reviewer audits commit
  │    ├─ approved → wait for wave merge step
  │    └─ issues_found
  │         ├─ retry_count < 2 → same implementer fix-loop
  │         └─ retry_count == 2 → mark failed_blocking, escalate at wave barrier
  │
  ├─ STATUS=blocked → mark blocked, do not abort wave siblings
  ├─ STATUS=failed_blocking → mark failed_blocking, do not abort wave siblings
  └─ STATUS=timed_out → mark timed_out, do not abort wave siblings

wave merge step (sequential, plan order)
  on FILES_CHANGED ⊄ declared writes:
    abort merge, mark wave contaminated, escalate
    
wave_quality_reviewer (after merge)
  ├─ approved → controller advances
  ├─ critical_blocking → block next wave, fix required
  └─ minor_deferred → write to followup.md, advance allowed
```

### Worktree isolation

- Each implementer runs in independent git worktree (existing `superpowers:using-git-worktrees` pattern)
- Worktree-level file isolation + `writes:` disjointness check + post-hoc FILES_CHANGED verify = three-layer protection
- Wave integration branch created by the runner from base commit at wave start; each task's worktree merges sequentially into wave branch in plan order
- Code quality reviewer sees the integrated wave-branch diff

## Capability registry + integration

### New capabilities

Added to `claude/capabilities/defaults.json`:

```json
{
  "wave_planning": "flow:flow-wave-planner",
  "wave_dispatch": "flow:flow-wave-runner"
}
```

**Why split**: planner concerns (decompose, cache, preview, doctor checks) are static and idempotent. Runner concerns (worktree, merge, retry, terminal state machine) are runtime and stateful. Different testability surfaces, different update cadences.

### New skills

| Skill | Path | Responsibility |
|-------|------|----------------|
| `flow-wave-planner` | `claude/skills/flow/flow-wave-planner/SKILL.md` | Read plan, run independence algorithm (mechanical first → LLM veto), write/reuse cache, emit waves[] |
| `flow-wave-runner` | `claude/skills/flow/flow-wave-runner/SKILL.md` | Receive wave + SHARED_ARTIFACTS, set up worktrees, dispatch paired implementer/spec-reviewer, sequential merge with FILES_CHANGED verify, dispatch code-quality reviewer |
| `flow-shared-artifacts` (static reference) | `claude/skills/flow/flow-wave-runner/SHARED_ARTIFACTS.md` | Glob list that forces wave to serial |

### Phase 2 orchestrator update

`claude/skills/flow/flow-phase2-execute/SKILL.md` adds ~30 lines covering:

- Wave-decomposition cache check + reuse (cache file: `.flow/tasks/<slug>/wave-decomposition.json`)
- Cache invalidation: plan file hash change OR base commit change
- Dispatch decision: wave.size==1 → `subagent_dispatch`, wave.size>1 → `wave_dispatch`
- Capability fallback: if `wave_dispatch` capability not found in user's registry → fallback to `subagent_dispatch` for entire plan (zero-regression degradation)
- Wave barrier: collect terminal states, apply default-block-on-failure policy, ask controller for explicit waive when blocked

### CLI surface

| Command | Purpose | When to use |
|---------|---------|-------------|
| `flow waves --preview` | Compute decomposition without executing; render cache as human-readable table | Before Phase 2 execute, sanity-check wave structure |
| `flow doctor` (extended) | New checks: tasks missing `writes:`, broad globs (`**/*`), writes touching SHARED_ARTIFACTS, stale wave cache | Anytime, especially before parallel runs |
| `flow doctor --suggest-writes` | (advisory only) Heuristic suggestions for `writes:` field on existing plans | When migrating old plans |

**No `flow:dispatch-wave` slash command** — no manual override path. All parallel dispatch must go through Phase 2 orchestrator (preserves invariants).

## Migration / backwards compatibility

| Scenario | Behavior |
|----------|----------|
| v0.6.x plan with no `writes:` fields | Auto-fallback to all-serial via `subagent_dispatch`. Zero regression. |
| User has old capability registry (no `wave_dispatch` cap) | Phase 2 detects missing cap, falls back to `subagent_dispatch` for entire plan. Zero regression. |
| User wants to migrate old plan to use parallel | Run `flow doctor --suggest-writes` for advisory output. Author confirms each `writes:` manually. **Never auto-write.** |
| Plan file edited after cache computed | Cache invalidates on plan_hash change. Next Phase 2 run recomputes waves. |
| Switch controller model (e.g., upgrade Opus version) | `controller_model` field in cache flags this; planner can opt to recompute or warn. |

## Out of scope (v0.7 will NOT do)

- **Cross-phase parallel** (Phase 1 and Phase 2 task overlap) — out of scope, deferred indefinitely
- **Subagent-to-subagent direct communication within a wave** — breaks independence premise, not supported
- **Auto-learning from historical plan executions** — insufficient data, deferred to v0.8+
- **Ruleset of "common independence patterns" baked into skill** — deferred, keep purely mechanical for now
- **Manual `flow:dispatch-wave` slash command** — would bypass safety; preview only

## Risks (residual after codex review)

| # | Risk | Mitigation in design | Residual |
|---|------|---------------------|----------|
| 1 | LLM concept check is non-deterministic — same plan may produce different waves on rerun | Cache decomposition keyed by plan_hash + base_commit + controller_model; cache reuse is the default | Different controller models or model upgrades will recompute; logged for audit |
| 2 | 7 in-flight subagents (cap=3) may hit Claude Code Task tool / rate limits | Default cap=3 chosen specifically to keep peak under 10; opt-in cap=4 documented as needing observation | Real rate-limit data only available after first parallel runs |
| 3 | Code-quality reviewer's view of merged wave diff may exceed context if all 3 tasks edit large files | Per-task SUMMARY in implementer return + reviewer fetches selective `git show` ranges | Large refactors may need wave size 1 manually; doctor flags broad glob patterns as warning |
| 4 | Sequential merge can fail mid-wave if task K introduces unexpected file write | FILES_CHANGED verification at each merge step catches contamination immediately | Wave abort = manual triage; cost is one wave's worth of subagent time wasted |
| 5 | Controller waive of `failed_blocking` requires sound dependency reasoning | Decision logged to `wave-decisions.log` with rationale; reviewable post-hoc | Trust in controller judgment is the limit; same as today's serial dispatch |
| 6 | Generated/shared files outside SHARED_ARTIFACTS list cause hidden races | Initial list covers common cases (lockfiles, registries, version files); expanded based on observed collisions | Discovery is reactive, not proactive |
| 7 | Author writes inaccurate `writes:` glob (too narrow → race; too broad → serial) | Post-hoc FILES_CHANGED verification catches "too narrow" cases; doctor flags "too broad" | Author still needs basic glob hygiene; trade-off accepted |

## Acceptance criteria

v0.7 is shippable when:

1. ✅ All v0.6.x plans run unchanged (full backwards compat)
2. ✅ A plan with declared `writes:` and 2+ disjoint tasks demonstrably runs them in parallel
3. ✅ A plan with overlapping `writes:` falls back to serial without errors
4. ✅ A plan touching SHARED_ARTIFACTS forces serial regardless of declarations
5. ✅ FILES_CHANGED verification correctly catches a task that wrote outside its declared `writes:`
6. ✅ Cache invalidates on plan edit and recomputes waves
7. ✅ Capability missing → graceful fallback to serial
8. ✅ `flow waves --preview` produces accurate decomposition output
9. ✅ `flow doctor` detects missing/broad writes
10. ✅ Real run on a 5+ task plan: at least one wave runs N>1 in parallel and integrates cleanly
11. ✅ Failed task properly blocks next wave by default; controller can explicitly waive
12. ✅ All existing v0.6.x tests pass; new tests cover wave planning, merge verification, fallback paths
