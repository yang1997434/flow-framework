## Problem statement

Flow Phase 2 (`execute`) currently dispatches implementer subagents **sequentially** via `superpowers:subagent-driven-development`. That skill's guidance line ("Never dispatch multiple implementation subagents in parallel — conflicts") is correct **for the single-task case**, but it forces all multi-task plans into serial execution even when tasks touch disjoint files.

v0.6.0 implementation took 7 sequential subagent waves; **5 of them were file-level disjoint** and could have run concurrently. Wall time roughly halves if those run in parallel.

v0.7 adds a **wave-based parallel dispatcher** that runs only proven-independent tasks concurrently, with conservative defaults. The controller (main Claude session) is the parallelism judge — authors write linear plans, the controller decomposes into waves at Phase 2 entry.

---

## Design philosophy

1. **Conservative by default** — any uncertainty about independence → serial. Match Bazel's hermeticity spirit, not Make's silent races.
2. **Mechanical proof primary, LLM judgment secondary** — file-write disjointness is the positive proof; LLM concept check can only veto (downgrade parallel→serial), never elevate (serial→parallel).
3. **Reproducibility over flexibility** — wave decomposition is cached and keyed by `plan_hash + base_commit + controller_model + planner_version + cap_used`. Same inputs → same waves.
4. **Verification at every boundary** — declared `writes:` is a hypothesis; the wave integration step verifies actual `FILES_CHANGED ⊆ declared writes`. Violation → wave contaminated, stop.
5. **Backwards compat = zero regression** — plans without `writes:` field auto-fallback to all-serial. Capability missing → fallback to existing `subagent_dispatch`. v0.6.x plans run unchanged.

## Architecture

### Phase 2 orchestrator flow

```
Phase 1 (plan)
  ↓ produces implementation_plan with optional `writes:` per task

Phase 2 (execute) — orchestrator
  Step A: read plan, invoke `wave_planning` capability
    → planner reads cache (.flow/tasks/<slug>/wave-decomposition.json)
    → if plan_hash + base_commit match: reuse cached waves
    → else: compute waves via independence algorithm, write cache
    → emit waves[]
  Step B: for each wave in waves[]:
    - if wave.size == 1 → invoke `subagent_dispatch` capability (existing path)
    - if wave.size  > 1 → invoke `wave_dispatch` capability (new flow-wave-runner)
  Step C: at wave barrier, controller (main session) judges:
    - all tasks reached terminal state?
    - any failed_blocking → MUST FIX (cannot be waived); fix-loop with implementer or escalate
    - any blocked / timed_out / cancelled → blocks next wave by default; controller may waive with logged rationale
    - critical_blocking from code-quality reviewer → must fix before advancing
    - else → advance to wave[k+1]

Phase 3 / Phase 4 — unchanged
```

### Plan schema extension

```yaml
# implementation_plan.yaml
tasks:
  - id: task-1-update-phase1-skill
    title: "Update Phase 1 SKILL.md to add Engineer hat"
    writes:                                # OPTIONAL, glob list — hard independence evidence
      - claude/skills/flow/flow-phase1-plan/SKILL.md
    reads:                                 # OPTIONAL, hint only — does not affect independence
      - docs/Skills-Phase映射.md
    description: |
      ...
```

Field semantics:
- `writes`: file globs the task plans to write. Absent → task is **strictly serial** (cannot join any wave).
- `reads`: file globs the task plans to read. **Hint only** — N tasks reading the same file does not block parallel.
- Plans without `writes:` field anywhere → entire plan runs all-serial (v0.6.x behavior preserved).

### Independence algorithm

**Contiguous-prefix policy**: plan order is the implicit dependency declaration. The planner forms each wave from the **contiguous prefix** of unprocessed tasks. As soon as a task cannot join the current wave, the wave is emitted and a new wave starts from that task. The planner never reorders tasks past a non-joiner.

```
input: ordered list of tasks (plan order = implicit dependency declaration)
output: list of waves (each wave is a list of tasks; concatenating preserves plan order)

remaining = tasks  # in plan order, mutable
while remaining:
  wave = [remaining.pop(0)]
  while remaining and len(wave) < CAP:
    next_task = remaining[0]
    if can_join_wave(next_task, wave):
      wave.append(next_task)
      remaining.pop(0)
    else:
      break              # contiguous-prefix: do not skip past a non-joiner
  emit wave
  
can_join_wave(t, wave):
  for w in wave:
    # Step 1: mechanical disjointness (must pass)
    if t.writes is None or w.writes is None:
      return False                        # missing writes → strict serial
    if globs_overlap(t.writes, w.writes):
      return False                        # writes overlap → serial
    if globs_overlap(t.writes, SHARED_ARTIFACTS) or globs_overlap(w.writes, SHARED_ARTIFACTS):
      return False                        # any wave member touching a shared artifact → forced serial
    if any glob in t.writes ∪ w.writes is bare '*' / '**' / '**/*':
      return False                        # too broad to verify disjointness → serial
    
    # Step 2: LLM concept check (veto only — can downgrade parallel → serial, never elevate)
    if controller LLM-judges "t and w conceptually coupled":
      return False                        # e.g., both modify same orchestrator behavior across files,
                                          #       or t exports a type w consumes,
                                          #       or t is docs reflecting code w writes
  return True

globs_overlap(globs_a, globs_b):
  # Two glob sets overlap if any concrete path matches a glob from each set.
  # Implementation: normalize each glob to a path prefix + pattern; check pairwise
  # whether any glob in A and any glob in B could match the same path. Use a standard
  # glob library (e.g., minimatch / fnmatch) for matching semantics. No negation patterns
  # allowed in writes: globs (banned at doctor time).
  ...
```

**Glob hygiene** (enforced by `flow doctor` AND by algorithm):
- No negation patterns (`!path`) in `writes:`
- No bare `*`, `**`, or `**/*` (too broad — algorithm rejects, doctor surfaces pre-execution)
- Globs must be repo-relative paths
- Generated outputs (e.g., `src/generated/**`, `*.snap`, `docs/api.generated.md`) **should be declared explicitly** in `writes:` if a task produces them — undeclared generated files trigger contamination at merge time

**Why mechanical-first**: codex consult flagged that LLM-only positive proof of independence is unsafe. Mechanical disjointness gives a deterministic positive answer; LLM only refines downward.

**LLM concept check signals** (controller veto on any of these):
- t introduces a new type, w consumes a similar type → contract race
- t and w both touch the same conceptual contract even in different files (e.g., capability registry semantics)
- t is documentation of behavior w implements → drift risk
- t and w both modify the orchestrator dispatch logic in different skills

### SHARED_ARTIFACTS list (initial)

Stored in `claude/skills/flow/flow-wave-runner/SHARED_ARTIFACTS.md`. Any task whose `writes:` glob has any overlap with these globs forces the entire wave to serial regardless of declared disjointness. Globs use `**/` recursive prefix to match nested locations (not just repo root):

```
**/package.json
**/package-lock.json
**/*.lock
**/yarn.lock
**/pnpm-lock.yaml
**/Cargo.lock
**/go.sum
**/go.mod
**/requirements.txt
**/poetry.lock
.flow/skills-map.md
claude/capabilities/*.json
VERSION
CHANGELOG.md
```

Extended over time as we observe collisions in practice. The static reference file is versioned with the planner skill so cache invalidation triggers when the list updates.

## Wave runtime (`flow-wave-runner` skill)

### Concurrency structure

```
Wave[k] = [task_1, ..., task_N]   where N ≤ CAP (default 3)

Time slot 1: 2N concurrent subagents (paired implementer + spec reviewer per task)
  paired_1: implementer_1 → spec_reviewer_1 → fix-loop (max 2 retries) → terminal
  paired_2: implementer_2 → spec_reviewer_2 → fix-loop (max 2 retries) → terminal
  ...
  paired_N: implementer_N → spec_reviewer_N → fix-loop (max 2 retries) → terminal
  
  (paired runs in parallel across N; within each paired, sequential)

Time slot 2: sequential merge into wave integration branch (controller)
  Wave[0] integration branch base = repo HEAD when execute starts
  Wave[k>0] integration branch base = HEAD of Wave[k-1] integration branch (after merge succeeded)
  
  for task in wave (plan order):
    pre_merge_sha  = current wave-branch HEAD
    git merge worktree-of-task → wave-branch
    post_merge_sha = current wave-branch HEAD
    
    # Per-task diff: only what THIS merge introduced, not cumulative across tasks.
    # Using <wave-base>..HEAD here would falsely flag earlier tasks' files
    # as contamination for later merges. Use pre/post sha pair instead.
    actual_changed = `git diff --name-only <pre_merge_sha>..<post_merge_sha>`
    
    verify: actual_changed ⊆ declared task.writes   # strict subset, no exceptions
    on git merge conflict → mark task contaminated, abort wave, escalate
    on subset violation    → mark wave contaminated, abort merge, escalate
    (rationale: any undeclared write — even expected ones like formatter output
     or generated files — must be added to the task's `writes:` glob explicitly.
     Strict subset is the safest invariant. Verification source must be git itself,
     not subagent-reported FILES_CHANGED, to prevent self-report drift.)

Time slot 3: 1 subagent
  code_quality_reviewer reads merged wave-branch diff (cross-task consistency)
  
Time slot 4: main controller (me)
  collect N status reports + 1 code-quality report
  judge: all terminal? critical issues fixed? integration coherent?
  → advance to wave[k+1] OR escalate
```

**Why sequential merge** (not octopus): codex flagged that even disjoint declared writes can collide on generated files (lockfiles, formatted output, snapshots). Sequential merge in plan order, with FILES_CHANGED verification per merge, catches contamination at the earliest task that produced it.

**Concurrency cap (CAP)**:
- Default: **3** (peak in-flight: 3 implementers + 3 spec reviewers + 1 code reviewer = 7)
- Opt-in: **4** via project config `claude/flow.config.json` → `parallel_dispatch.cap: 4` (peak: 9)
- N>CAP independent tasks → chunked into ⌈N/CAP⌉ waves of size ≤CAP

### Subagent return contracts

Each subagent must return ONLY these fields. Total budget enforced in subagent prompt.

**Implementer** (~600 token cap):
```
STATUS: done | done_with_concerns | blocked | timed_out
COMMIT_SHA: <full sha or "none">
SUMMARY: 3-5 bullets — what changed and why
CONCERNS: [] or short list
NEXT: 1-2 lines — integration notes for controller
```

Notes:
- Implementer **does not** report FILES_CHANGED — the controller derives the actual changed files via `git diff --name-only <pre_merge_sha>..<post_merge_sha>` per task at merge time (subagent self-report is not authoritative for verification; per-task delta avoids cross-task false positives).
- `failed_blocking`, `failed_nonblocking_waived`, and `cancelled` are **controller-derived states**, not implementer self-reports. Controller assigns them during wave-barrier evaluation.

**Spec reviewer** (~400 tokens):
```
VERDICT: approved | issues_found
COMMIT_SHA: <reviewed sha>
ISSUES: [] or list of {file, line, severity: critical|important|minor, fix_required: bool}
RETRY_COUNT: 0 | 1 | 2
```

**Code quality reviewer** (~600 tokens, runs once per wave at integration):
```
WAVE_VERDICT: approved | critical_blocking | minor_deferred
CRITICAL_ISSUES: []  # block next wave
MINOR_ISSUES: []      # write to .flow/tasks/<slug>/followup.md
CROSS_TASK_OBSERVATIONS: 2-3 bullets — duplication, style drift, contract drift
```

Diffs are NOT in returns. Controller fetches via `git show <sha>` only when integration check requires it.

### Terminal state taxonomy

| State | Meaning | Source | Blocks next wave by default? | Controller can waive? |
|-------|---------|--------|------------------------------|----------------------|
| `done` | Task complete, spec approved, no concerns | implementer | No | n/a |
| `done_with_concerns` | Task complete with concerns; spec reviewer still ran and approved | implementer | No (concerns logged) | n/a |
| `blocked` | Implementer cannot proceed, no commit | implementer | **Yes** | **Yes** (controller waives if no downstream task depends) |
| `timed_out` | Subagent exceeded wall-clock budget | implementer | **Yes** | **Yes** |
| `failed_blocking` | Spec reviewer found unfixable issues after 2 retries; impacts later waves | controller-derived (after spec review fix-loop exhausted) | **Yes** | **No** — must be fixed before advancing |
| `failed_nonblocking_waived` | A `blocked` / `timed_out` state that controller explicitly waived | controller-derived (waiver action) | No | Already waived |
| `cancelled` | Manually cancelled by controller | controller-derived | **Yes** | **Yes** (waive = no further action needed) |

Wave barrier requires every task in `Wave[k]` to be in some terminal state. Default policy: any non-success terminal state blocks `Wave[k+1]`, with two distinct resolution paths:
- `failed_blocking` — **MUST be fixed** (cannot be waived). Controller dispatches fix subagent or escalates to user.
- `blocked` / `timed_out` / `cancelled` — controller may either fix OR explicitly waive (downgrade to `failed_nonblocking_waived`) with rationale logged to `.flow/tasks/<slug>/wave-decisions.log`.

### Failure / retry tree

```
implementer returns
  ├─ STATUS=done OR done_with_concerns → spec_reviewer audits commit  (review runs in BOTH cases)
  │    ├─ approved → task ready for wave merge step
  │    └─ issues_found
  │         ├─ retry_count < 2 → same implementer fix-loop
  │         └─ retry_count == 2 → controller marks task as failed_blocking
  │                                (cannot waive; must be fixed before next wave)
  │
  ├─ STATUS=blocked → mark blocked, do not abort wave siblings  (waivable)
  └─ STATUS=timed_out → mark timed_out, do not abort wave siblings (waivable)

wave merge step (controller, sequential, plan order)
  for task in wave (plan order, only tasks with terminal-and-mergeable state):
    git merge task.worktree → wave-branch
    on git conflict → controller marks task contaminated, abort wave, escalate
    actual_changed = git diff --name-only <wave-base>..HEAD
    if actual_changed ⊄ task.writes:
      abort wave, mark wave contaminated, escalate (require updated `writes:` and recompute waves)
    
wave_quality_reviewer (after all merges succeed)
  ├─ approved → controller advances to wave[k+1]
  ├─ critical_blocking → block next wave, fix required (analogous to failed_blocking)
  └─ minor_deferred → write to .flow/tasks/<slug>/followup.md, advance allowed
```

### Worktree isolation

- Each implementer runs in independent git worktree (existing `superpowers:using-git-worktrees` pattern)
- Worktree-level file isolation + `writes:` disjointness check + post-hoc per-task `git diff --name-only <pre_merge_sha>..<post_merge_sha>` verification = three-layer protection
- Wave integration branch base policy:
  - **Wave[0]** branches from repo HEAD at Phase 2 start
  - **Wave[k>0]** branches from HEAD of Wave[k-1] integration branch (after successful merge)
- Each task's worktree merges sequentially into the wave integration branch in plan order; `git diff --name-only` verifies strict subset of `writes:` after each merge
- Code quality reviewer sees the integrated wave-branch diff (against the wave's base, not against repo HEAD)

## Capability registry + integration

### New capabilities

Added to `claude/capabilities/defaults.json`:

```json
{
  "wave_planning": "flow:flow-wave-planner",
  "wave_dispatch": "flow:flow-wave-runner"
}
```

**Why split**: planner concerns (decompose, cache, preview, doctor checks) are static and idempotent. Runner concerns (worktree, merge, retry, terminal state machine) are runtime and stateful. Different testability surfaces, different update cadences.

### New skills

| Skill | Path | Responsibility |
|-------|------|----------------|
| `flow-wave-planner` | `claude/skills/flow/flow-wave-planner/SKILL.md` | Read plan, run independence algorithm (mechanical first → LLM veto), write/reuse cache, emit waves[] |
| `flow-wave-runner` | `claude/skills/flow/flow-wave-runner/SKILL.md` | Receive wave + SHARED_ARTIFACTS, set up worktrees, dispatch paired implementer/spec-reviewer, sequential merge with FILES_CHANGED verify, dispatch code-quality reviewer |
| `flow-shared-artifacts` (static reference) | `claude/skills/flow/flow-wave-runner/SHARED_ARTIFACTS.md` | Glob list that forces wave to serial |

### Phase 2 orchestrator update

`claude/skills/flow/flow-phase2-execute/SKILL.md` adds ~30 lines covering:

- Wave-decomposition cache check + reuse (cache file: `.flow/tasks/<slug>/wave-decomposition.json`)
- Cache invalidation (any of these mismatching the cache value triggers recompute):
  - `plan_hash` (plan file content changed)
  - `base_commit` (repo HEAD moved)
  - `controller_model` (model upgraded — different judgment surface)
  - `planner_version` (planner skill itself changed semantics)
  - `cap_used` (project config changed concurrency cap)
- Dispatch decision: wave.size==1 → `subagent_dispatch`, wave.size>1 → `wave_dispatch`
- Capability fallback (zero-regression degradation):
  - `wave_planning` capability missing → cannot decompose; fallback to all-serial via `subagent_dispatch` for the entire plan
  - `wave_dispatch` capability missing → can decompose for preview, but actual execution falls back to all-serial via `subagent_dispatch`
  - Either fallback logged with reason at Phase 2 entry
- Wave barrier: collect terminal states, apply default-block-on-failure policy, ask controller for explicit waive when blocked / timed_out / cancelled (`failed_blocking` cannot be waived; must be fixed)

## Migration / backwards compatibility

| Scenario | Behavior |
|----------|----------|
| v0.6.x plan with no `writes:` fields | Auto-fallback to all-serial via `subagent_dispatch`. Zero regression. |
| User has old capability registry (no `wave_dispatch` cap) | Phase 2 detects missing cap, falls back to `subagent_dispatch` for entire plan. Zero regression. |
| User wants to migrate old plan to use parallel | Run `flow doctor --suggest-writes` for advisory output. Author confirms each `writes:` manually. **Never auto-write.** |
| Plan file edited after cache computed | Cache invalidates on plan_hash change. Next Phase 2 run recomputes waves. |
| Switch controller model (e.g., upgrade Opus version) | `controller_model` field in cache flags this; planner can opt to recompute or warn. |

## Out of scope (v0.7 will NOT do)

- **Cross-phase parallel** (Phase 1 and Phase 2 task overlap) — out of scope, deferred indefinitely
- **Subagent-to-subagent direct communication within a wave** — breaks independence premise, not supported
- **Auto-learning from historical plan executions** — insufficient data, deferred to v0.8+
- **Ruleset of "common independence patterns" baked into skill** — deferred, keep purely mechanical for now
- **Manual `flow:dispatch-wave` slash command** — would bypass safety; preview only

## Acceptance criteria

v0.7 is shippable when:

1. ✅ All v0.6.x plans run unchanged (full backwards compat)
2. ✅ A plan with declared `writes:` and 2+ disjoint tasks demonstrably runs them in parallel
3. ✅ A plan with overlapping `writes:` falls back to serial without errors
4. ✅ A plan touching SHARED_ARTIFACTS (via glob overlap, not just membership) forces serial regardless of declarations
5. ✅ Verification using `git diff --name-only` (NOT subagent-reported FILES_CHANGED) correctly catches a task that wrote outside its declared `writes:`
6. ✅ Cache invalidates on **any** of: `plan_hash`, `base_commit`, `controller_model`, `planner_version`, `cap_used` change — and recomputes waves
7. ✅ Capability fallback works for both missing `wave_planning` and missing `wave_dispatch` (zero-regression to all-serial)
8. ✅ `flow waves --preview` produces accurate decomposition output with rationale per pair
9. ✅ `flow doctor` distinguishes **fatal-for-parallel** issues (broad globs, missing writes, SHARED_ARTIFACTS overlap) from **warnings** (e.g., narrow-but-suspect globs)
10. ✅ Planner does not violate plan order (contiguous-prefix policy verified — no task runs before an earlier task that couldn't join)
11. ✅ LLM veto can downgrade mechanically-disjoint tasks to serial; LLM cannot promote overlapping tasks to parallel (one-way semantics)
12. ✅ `failed_blocking` cannot be waived (must be fixed); `blocked` / `timed_out` / `cancelled` can be waived; waiver logged with rationale to `.flow/tasks/<slug>/wave-decisions.log`
13. ✅ Sequential-merge git conflict and undeclared-write contamination both fail cleanly with a logged reason and the contaminating task identified
14. ✅ Wave[k>0] integration branch correctly bases on Wave[k-1] HEAD (not original repo HEAD)
15. ✅ All existing v0.6.x tests pass; new tests cover wave planning (algorithm correctness, contiguous-prefix invariant, glob overlap, SHARED_ARTIFACTS), merge verification, terminal state machine, and fallback paths
16. (Smoke / release validation, not a unit-test acceptance) Real run on a 5+ task plan: at least one wave runs N>1 in parallel and integrates cleanly
