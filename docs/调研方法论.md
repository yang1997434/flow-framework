# AI 编程调研方法论

> 灵感来源：[[Trellis调研]]——吸收它的核心方法论，**用现有工具实现**，不依赖 trellis 安装。

## 核心问题

AI 编程任务里"调研"经常做得很糟糕：
- 在主对话里 inline 多次 `WebFetch` / `WebSearch`，把原始 HTML/JSON 灌进 context，烧 token
- 调研结论只留在聊天记录里，下次 session 全丢
- 让用户自己发明候选方案（"你想用哪个状态管理库？"），把推理负担甩给用户
- "代码里写了什么样子？"——明明仓库就在，硬要问

这套方法论解决以上问题。

## 4 大原则

### 1. 文件持久化，对话不持久

> Conversations get compacted; files don't.

**调研产出必须落到文件**，不能只留在主对话里。理由：context 窗口会被 compact / clear，下一个 session 就丢了。

**操作**：
- 每个调研主题一个 `.md` 文件（见下面"输出位置"决策）
- 主对话里只放**一句话 takeaway + 文件链接**

### 2. Sub-agent 隔离，不污染主对话

主 session 自己 `WebFetch` 10 次会把原始网页内容灌进主 context（烧 token、污染思路、context 后段被截）。

**正确做法**：派 sub-agent 做调研：
- sub-agent 有自己独立的 context 窗口
- 它做完研究，**只把"文件路径 + 一句话总结"返回给主 agent**
- 主 agent 之后再按需 `Read` 看详情

```text
❌ 反模式：
   主 session：WebFetch(A) → WebFetch(B) → WebSearch(q1)
              → WebSearch(q2) → ... → 写 research.md
   → 主 context 已被 50 KB 原始网页污染

✅ 正确模式：
   主 session：Agent(subagent_type="general-purpose",
                     prompt="调研 A，写到 research/topic-a.md")
            ‖ Agent(subagent_type="general-purpose",
                     prompt="调研 B，写到 research/topic-b.md")
            ‖ Agent(subagent_type="Explore",
                     prompt="找代码里 X 的引用，写到 research/x-refs.md")
   ← 三个 sub-agent 并行
   → 主 session 拿到 3 个一行 summary + 3 个文件路径
   → 按需 Read 看详情
```

**Agent 选型**：

| Sub-agent | 用途 |
|-----------|------|
| `Explore` | 在代码库里搜索符号 / 文件 / 引用（**不**适合 review、设计、跨文件分析）|
| `general-purpose` | 通用调研，可用全部工具，适合外部资源 + 多步任务 |
| `Plan` | 写实施计划，不写代码 |
| `pr-review-toolkit:*` | 各类专项 review |

### 3. Research-first for technical choices

涉及"选哪个库 / 框架 / 协议 / 模式"的问题，**先调研，再给用户 2-3 个具体选项**。**不要让用户自己发明候选方案**。

```text
❌ 反模式："你想用哪种状态管理库？"
   → 用户也不知道，被迫自己研究

✅ 正确模式：
   1. 派 sub-agent 调研：当前生态主流状态管理方案
   2. 自己给 3 个选项（Zustand / Jotai / Redux Toolkit），各带 pros/cons
   3. 问用户："Approach A / B / C 选哪个？"
```

### 4. Action before asking

能从代码 / 文档 / config 推出来的问题，**自己看，不要问用户**。

| 反模式 | 正确做法 |
|--------|---------|
| "代码里写了什么样子？"（仓库就在）| 自己 `grep` / `Read` |
| "我应该搜一下吗？"（meta 问题）| 直接搜 |
| "你能粘贴代码我才能继续吗？"| 自己读 |

只问真正的 **blocking** 或 **preference** 问题。

## 操作模板

### 起一个调研型任务

```text
1. 用 planning-with-files 起任务
   /plan 或 invoke planning-with-files:planning-with-files
   → 创建 task_plan.md, findings.md, progress.md

2. 把调研问题拆成独立子题（每题一个 sub-agent 可承担的颗粒度）
   写到 task_plan.md

3. 一条消息派多个 sub-agent 并行
   每个 sub-agent prompt 必须包含：
     - 调研目标（一句话）
     - 输出文件路径（绝对路径）
     - 输出格式约束（结构 / 字段 / 长度上限）
     - 已知约束 / 不要做什么

4. 等所有 sub-agent 返回
   读它们的 takeaway，按需 Read 文件细节

5. 在 findings.md 收敛
   只放 takeaway + 文件链接，不要复制完整内容

6. 决策落 ADR-lite（见下面"决策记录格式"）
```

### Sub-agent 调研 prompt 模板

```markdown
任务：调研 <具体问题>

输出要求：
- 文件路径：<绝对路径，如 /path/to/findings/topic.md>
- 文件格式：
  ## TL;DR（不超过 3 行）
  ## 候选方案（2-3 个，各带 pros/cons）
  ## 我们 stack 的兼容性
  ## 推荐 + 理由
  ## 参考链接

约束：
- 字数上限 800
- 不读 <某个不相关目录>
- 已知 <某个前置事实>，不需要再调研

返回给我（主 agent）：仅 1 行 summary + 文件路径，不要把全文返回。
```

### 调研主题拆分（关键技巧）

错误的拆分：粒度太大，sub-agent 要做 5 件事

```text
❌ "调研所有前端状态管理方案，给推荐"
   → sub-agent 自己也得拆分，结果质量参差
```

正确的拆分：每个主题独立 + 可并行

```text
✅ 拆成 4 个独立 sub-agent：
   1. research/state-mgmt-comparison.md — Zustand/Jotai/Redux 横向特性
   2. research/state-mgmt-bundle-size.md — 各方案打包后的体积
   3. research/state-mgmt-our-stack.md — 我们已有依赖的兼容性
   4. research/state-mgmt-migration-cost.md — 从当前方案迁移的工作量

→ 4 sub-agent 一条消息并行派
→ 主 agent 拿 4 个 takeaway，自己合成推荐
```

### 决策记录格式（ADR-lite）

调研收敛到决策时，**必须写下来**（落到 task 的 PRD / findings.md / vault 都行）：

```markdown
## Decision (ADR-lite)

**Context**: 为什么需要这个决策（背景 + 触发事件）
**Decision**: 选了什么 + 拒绝了什么
**Consequences**:
  - 短期代价（如：迁移期 X 周）
  - 长期收益（如：解决 Y 问题）
  - 反悔成本（如：Z 周可回滚 / 不可逆）
**Revisit triggers**: 什么情况下重新评估这个决策
```

## 输出位置决策

| 内容类型 | 写到哪 | 理由 |
|---------|-------|------|
| 单次任务的调研中间产物 | `<task_dir>/research/*.md` 或 `findings.md` | 任务级，不需要长期保存 |
| 决策结论 + ADR-lite | 任务的 PRD / findings.md **+** vault 一篇笔记 | 任务级 + 跨项目复用 |
| 通用方法论 / 工具评估 | vault `~/data/knowledge-base/` | 跨项目长期参考 |
| 调研用到的代码片段 / 配置 | vault 同篇笔记的代码块 | 一起沉淀 |
| 临时 sub-agent 输出 | `/tmp/<topic>.md` 或任务目录 | 用完即弃 |

## 与现有工具的映射

| Trellis 概念 | 我用什么实现 |
|------------|------------|
| `trellis-research` sub-agent | `Agent(subagent_type="general-purpose")` 或 `Explore` |
| `{task}/research/*.md` 持久化 | `planning-with-files` 的 `findings.md` 或 vault 笔记 |
| PRD 的 `Research References` 段 | `planning-with-files` 的 `task_plan.md` 里链接 findings.md |
| `Decision (ADR-lite)` | 同上，或 vault 笔记 `## Decision` 段 |
| 多 sub-agent 并行 | 一条消息里多个 `Agent` tool calls |
| `trellis-brainstorm` skill | `superpowers:brainstorming` skill |
| `trellis-implement` sub-agent | 主 session 直接写 + `superpowers:test-driven-development` |
| `trellis-check` sub-agent | `pr-review-toolkit:code-reviewer` 或 `/codex review` |

## 反模式清单（硬避免）

- ❌ 主对话里 `WebFetch` 5 次以上不停
- ❌ 调研结论只留在聊天，没落文件
- ❌ 一个 sub-agent 派出去做 5 件事（颗粒度过大）
- ❌ 让用户自己想"你要 A 还是 B 还是 C"，但 A B C 还没研究清楚
- ❌ 派 sub-agent 时不给输出文件路径（结果它把全文塞回主 context）
- ❌ 调研做完了不收敛，留一堆零散文件没合成结论

## 何时**不**走这套流程

不是所有任务都要走。**简单任务硬走流程是反生产**。

| 跳过流程的条件（满足任一）| 说明 |
|------|------|
| Trivial 改动 | 改 typo / 单行 fix / 显然的小修补 |
| 探索性问题 | "X 是什么？""Y 怎么用？"——直接答 |
| 单次性查询 | 一两次工具调用就能拿到答案，不需要 sub-agent |
| 用户明确说"别走流程" / "直接改" / "小修一下" | 用户明令，本轮跳过 |

判断标准：**调研深度是否值得 sub-agent 启动开销**。Sub-agent 启动有约 5-30 秒开销 + 单独 context 成本，对于 30 秒能答完的问题就是浪费。

## 落地清单

下次做调研时，按这个 checklist 自检：

- [ ] 有没有把调研问题落到文件？（不只是聊天）
- [ ] 主对话有没有 inline 3 次以上 `WebFetch/WebSearch`？（如有，应改派 sub-agent）
- [ ] 有没有让用户从空白发明选项？（如有，应先调研再给选项）
- [ ] 决策有没有写 ADR-lite？
- [ ] sub-agent prompt 有没有指定输出文件路径？
- [ ] 多个独立调研主题有没有并行派？
- [ ] 完成时有没有把可复用结论沉淀到 vault？
