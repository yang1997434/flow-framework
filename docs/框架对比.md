# Flow vs 行业 AI 编码框架对比

> 把我们的 [[编码框架|Flow]] 放在行业坐标系里，看它**像什么、不像什么、独特点在哪**。
> 本对比基于 [[Trellis调研]] + 行业阈值调研（`/tmp/flow-research/industry-thresholds.md`）+ 记忆 promotion 调研（`/tmp/flow-research/memory-promotion.md`）。

## 框架坐标系

行业里"AI 编码"工具大致分四类，各解决不同层级的问题：

```
┌──────────────────────────────────────────────────────────┐
│ Layer 1: Agent harness（agent 跑在哪）                   │
│   Claude Code / Cursor / Continue / Aider / Codex        │
│   → 提供 LLM + 工具 + UI/CLI 的整合环境                   │
└──────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────┐
│ Layer 2: Multi-agent orchestration（多 agent 怎么协同）   │
│   CrewAI / AutoGen / LangGraph / smolagents / Mem0 / Letta│
│   → 提供"派几个 agent / 怎么交流 / 记忆怎么管"的框架     │
└──────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────┐
│ Layer 3: Project workflow（每个项目怎么用 AI 写代码）     │
│   Trellis / Cline Memory Bank / Roo Code / Pi Agent      │
│   → 提供"项目里 spec/tasks/journals 文件怎么组织"         │
└──────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────┐
│ Layer 4: Personal workflow（个人怎么和 AI 协作）          │
│   Anthropic memory tool patterns / Devin DiD model       │
│   → 提供"长期任务 / 跨 session 记忆 / 个人偏好"的模板    │
└──────────────────────────────────────────────────────────┘
```

**Flow 的定位**：**Layer 3 + Layer 4 的交叉**——**项目内编排 + 个人长期记忆**。它跑在 Layer 1（Claude Code）之上，不和 Layer 2 框架（CrewAI 等）竞争。

## Flow 真正独特的 5 个点

行业里**没有**框架同时具备这 5 个，这是 Flow 的差异化主张：

### 1. 显式凭据规约 + Staleness check

行业现状：所有调研过的工具**都没有** staleness check（仅 Trellis 有 spec drift 检测）。**所有工具都没有显式凭据规约**——它们假设你不会乱写。

Flow 提议：
- vault / `.flow/` committed 文件**绝不接受凭据**（命令级 grep 验证）
- 凭据走 `~/.flow/credentials.local`（chmod 600）/ 环境变量 / per-machine auto-memory
- 引用方式：`credentials_ref: <path>` 字段

实测验证（[[infra/README]]）：迁移 6 条混着密码的现有记忆时，grep 自检捕获到了凭据，**0 泄露**。

### 2. 踩坑系统作为一等公民 + trigger_paths 自动加载

行业现状：踩坑信息散落在各处——Trellis 在 spec/、Cursor 在 rules/、Claude Code 在 feedback type memory。**没有专门的"踩坑库"机制**。

Flow 提议：
- 独立的 `pitfalls/` 子树（项目级 / 跨项目）
- 标准 schema：Symptom / Root cause / Fix / Prevention / Why
- `trigger_paths` 字段——任务启动扫 prd.md，匹中相关 pitfalls 自动加载

实测验证（[[catbus-server-argparse-host]]）：5 分钟可以预防 30 分钟的重复踩坑。

### 3. Token 路由（任务级模型梯度）

行业现状：CrewAI / AutoGen 等都把模型选择留给用户全局配置；**没有"任务复杂度→自动选模型"的内置机制**。Aider 的 architect+editor 模式是个例外，但只是 2 角色不是任务级梯度。

Flow 提议：
- Triage Haiku 4.5 / 主 session Opus 4.7 1M / Implement Opus / Research Sonnet 4.6 1M / Verify Sonnet
- 总账：相比"全 Opus"省 ~2-3x，质量不打折

### 4. 跨模型异源审查（Codex / GPT-5.5）

行业现状：Aider 的 architect 是同模型不同 prompt；**没有框架内置"跨模型家族异源 review"**。Anthropic 自己的 multi-agent research system 是同 Anthropic 模型梯度。

Flow 提议：
- Phase 3 Finish 必触发条件下调 `/codex review`（gstack）
- Phase 2 卡住时 `/codex challenge`
- Phase 1 高反悔成本决策时 `/codex consult`

理由：Claude 自审 Claude 有结构性盲点。GPT 训练数据 / 推理偏向不同，能戳中 Claude 的失误。

### 5. 远程 SSH 工作流的显式适配

行业现状：所有调研工具都假设**本地开发**。SSH 远程工作场景几乎没人专门设计。

Flow 提议：
- `.flow/` 全相对路径
- 凭据规约（见 1）
- GUI-free 核心，可选层（impeccable / browser）opt-in
- `config.local.yaml` 标识 machine_id 防镜像同步冲突

## Flow 的薄弱 / 待验证（诚实标注）

| 维度 | 现状 | 风险 |
|------|------|------|
| **成熟度** | v0.1 草案 + 1 个 dogfood 任务实测 | 大量未验证的设计 |
| **Slash command 实现** | 只在文档里描述，未实现 | 上手要先建 `~/.claude/commands/flow/*` |
| **量化阈值依据** | engineering judgment（行业也没好数据）| 数字可能偏，需用真实任务调 |
| **Conflict detection 实现** | 设计是 pre-flight Sonnet check，未实测 | 可能误报 / 漏报 |
| **Team mode** | 设计了 task 锁，未实现 | 多人场景未压测 |

调研里学到的可以借鉴的点：

| 来源 | 借鉴点 | 已纳入 Flow？ |
|------|-------|-------------|
| **Anthropic Research** | 1 / 2-4 / 10+ subagent 梯度 | ✅ Phase 2 dispatch shape 同此 |
| **arxiv 2512.08296** | 单 agent baseline > 45% 时多 agent 反而降效；>10 工具退化单 agent | ✅ v0.2 加 escape hatch |
| **arxiv 2503.15703** | Amdahl's Law：specialize when parallelizability < N | ✅ "≥3 模块" 替代 "≥3 文件" |

| **Mem0 ADD/UPDATE/DELETE/NOOP** | similarity threshold pipeline 防重复 | ⚠️ 待 v0.2 考虑 |

| **Cursor Rules 路径限定 (`applyTo`)** | 按文件路径加载 rules | ✅ pitfalls 的 trigger_paths 同此思路 |

## 替代选择树

如果你看完上面想"我到底用哪个"：

```
你需要的是 ── ─ ─ ─ ─ ─ ─ ─ ─ → 推荐
"快速上手 + 单文件项目"     → Aider（轻量，零配置）
"团队仓库 + 多 agent 工具"   → Trellis（多平台 spec 共享）
"复杂多 agent 编排"         → CrewAI / AutoGen / LangGraph
"零配置 SaaS 体验"          → Cursor / GitHub Copilot
"个人长期任务 + 跨 session"  → Flow（我们的）
"自主代理 / 不参与编码"     → Devin
"私有部署 + 团队 RAG"       → Tabnine Enterprise / Cody Enterprise
```

Flow 是给**"个人为主 + 长期项目 + 重视记忆沉淀 + 经常远程 SSH 工作 + 愿意维护框架"**的用户。**不**适合：
- 需要 SaaS 零运维体验的用户
- 团队需要现成 PaaS 的场景（用 Trellis 更好）
- 只想要 IDE 补全的用户（用 Cursor / Copilot）

## 反思：要不要存在？

Flow 的根本问题：**它的所有独特点都可以拆出来单独用**：
- 凭据规约 → 一条 grep hook + ~/.flow/credentials.local 约定
- 踩坑系统 → 一个 vault `pitfalls/` 目录 + schema
- Token 路由 → Agent 工具 model 参数 + 一份 routing 表
- Codex 异源 → gstack `/codex` + 触发规则文档
- SSH 适配 → 几条最佳实践

把它们**编排成一个体系**才是 Flow 的真正价值。如果你只用其中 1-2 个，**完全不需要"装"Flow**——直接用现有工具就行。

→ 这意味着 Flow 的成败**取决于 Phase 编排在真实项目里好不好用**——这是 v0.2 之后真实任务才能验证的。
